# Seam A 프론트 배선 — 라이브 챗 런타임 + 대화 영속화

날짜: 2026-07-03 · 브랜치: `import-credigraph` · 지휘문서: [260702_마스터설계_ABC_워크스트림_분리실행.md](260702_마스터설계_ABC_워크스트림_분리실행.md)

## 한 줄 요약
프론트 챗을 사전 스크립트 **결정적 재생(목업)** 에서 실제 Upstage 추론(**Seam A `/api/chat`**)으로 배선하고(재생/라이브 **토글 병존**), 그 라이브 대화가 저장될 곳 — `public.conversations` — 을 신설·라이브 적용해 **관리자 하차장 목록·세무사 코멘트 원문·RAG write-path** 의 데이터 원천을 확보. 전 계층(계약·타입·RLS·컴파일) 검증 완료, 최종 브라우저 렌더는 육안 확인 잔여.

## 이 세션의 선행 — 프로세스 플로우 점검
사용자가 설계한 6단계 운영 흐름을 프로토타입과 대조([[project_operational_flow]] 신설):
1. 사장님 챗→Upstage 답변 — ✗ 목업(재생)이라 **배선 필요** → 이 세션에서 해결
2. 관리자 하차장 = 챗 목록(생성시간·종류·소유자 정렬, 제목 검색, 100 페이징) — ◑ 현재 엑셀 업로드 화면 + 요약만. **재편 필요(다음 태스크)**
3. 목록 클릭→읽기전용 상세→[일감 생성]·체크박스 배치, **엑셀 제거** — ◑ 배치는 `?conversationIds=` 로 이미 연결, 체크박스 UI 를 목록으로 이전 + 엑셀 삭제 필요
4. 일감 배정→세무사 문장 코멘트 — ✅ 구축됨(연결만)
5. 검사실 유효/무효·이의·기여 카운트 — ✅ 구축됨
6. 확정→바로 RAG(포장실 생략 OK)·정산 % 배분 — ◑✗ **finalize→RAG ingest 미배선(B 잔여)**, 정산 수익% 필드 없음, 기여의 RAG 존속 연동 없음

결론: 4·5 는 완성, **linchpin = 라이브 대화 영속화**(2·6·세무사 원문·RAG 가 모두 이걸 전제). 이 세션이 그 linchpin을 놓음.

## 산출물

**프론트 (신규):**
- `services/chat.ts` — Seam A `/api/chat` 원격 호출. `NEXT_PUBLIC_API_BASE` 소비, 요청 `{conversationId,occupation,history,userInput}`, 응답을 `conversation-schema`(Zod)로 검증(계약 불일치 즉시 노출). `?rag=false` A/B 스위치. `isRemoteChatConfigured` export.
- `lib/runtime/remote-chat-store.ts` — 라이브 대화 Zustand 스토어(conversationId·occupation·ownerId·ownerLabel·createdAt·messages·isRunning·error). 순수 store.
- `lib/runtime/remote-chat-send.ts` — 전송 액션(런타임 onNew + starter 공유 단일 진입점). user 노출→`/api/chat`→assistant 노출→**Supabase 영속화(best-effort)**.
- `lib/runtime/use-remote-runtime.ts` — assistant-ui `useExternalStoreRuntime` 래핑. onNew 가 스크립트 진행 대신 composer 텍스트를 캡처해 실제 추론 호출. 렌더는 재생 경로(thread.tsx external store 바인딩)와 동일 재사용.
- `components/chat/remote-chat-experience.tsx` — 라이브 챗 경험(진입 시 conversationId·createdAt 발급, starter=실제 첫 질문 전송, error 배너).
- `services/conversation.ts` — `persistLive(snapshot)`: 라이브 메시지를 유효 Conversation payload 로 빌드(order 재부여·프레임워크 수집·제목 자동생성) + `public.conversations` upsert. `buildLivePayload`·`deriveTitle` export.

**프론트 (수정):**
- `components/chat/chat-experience.tsx` — 셀렉터로 재편. 상단 **재생/라이브 토글**(ModeToggle). 기존 재생 로직은 `ReplayChatExperience` 로 보존(무손상). 기본 모드 = `NEXT_PUBLIC_CHAT_MODE`(미설정 시 replay).
- `.env.local` — `NEXT_PUBLIC_API_BASE=http://localhost:8787`(8000→8787, main.py 정합), `NEXT_PUBLIC_CHAT_MODE=replay` 추가.

**DB (마이그레이션 0005 — 라이브 적용):**
- `supabase/migrations/0005_conversations.sql` — `public.conversations`.
  - 목록·정렬 1급 컬럼(id·occupation·tax_category·**title**·**owner_id**·owner_label·source·status·turn_count·**created_at**·updated_at) + 전체 대화 `payload jsonb`(conversation-schema 원형).
  - 인덱스: created_at desc·occupation·owner_id·status(하차장 정렬).
  - RLS(0002 컨벤션 재사용): `conversations_owner`(소유자 CRUD, owner_id=current_domain_id), `conversations_staff_read`(admin·auditor 열람), `conversations_admin_all`.
  - Realtime publication 추가(하차장 실시간).

## 검증
- **백엔드 라이브 계약**: `/api/chat` clinic 실호출 → HTTP 200(13.9s), assistant 7세그(context/rule_statement/application/conclusion/evidence_request/caveat/follow_up) + uiBlocks[verdict_card, evidence_checklist], verdict=조건부, meta.engine=clinic_expense_engine, **ragHits=5**, ragCaseRefs 5건. 프론트 Zod 기대와 정합.
- **프론트**: `tsc --noEmit` 0 에러 · `eslint` 0 · `next dev`(Turbopack) 컴파일 OK, `/login`·`/chat/clinic` 200 서빙.
- **0005 라이브 적용**(Tokyo `credigraph`, psycopg 직접 + schema_migrations 0005 기록): 테이블·12컬럼·RLS 3정책·Realtime 포함·count=0(빈 출발) 확인.
- **RLS write path**(프론트와 동일 경로 — anon 클라이언트 + owner JWT): owner_id="viewer" insert → **201**(한글 보존), 잘못된 owner_id → **403**, admin·auditor 열람 → **각 1행**. 테스트 행 정리 후 count=0.
- **브라우저 E2E (사장님 계정, 육안 확인 완료)**: `/chat/clinic` 라이브 토글 → starter → Upstage 실응답 렌더(세그먼트 + framework 배지 + citations + **verdict 카드(전부인정)** + 필요증빙 체크리스트). `public.conversations` 에 **1행 적재**(occupation=clinic, owner=viewer/사장님, turn_count=2, source=live, payload messages=2). 영속화까지 라이브 확인.

### 버그 수정 (배선 중 발견)
- **`framework: null` 계약 불일치** — pydantic 이 `Optional` 필드를 `null` 로 직렬화하나 프론트 Zod `.optional()` 은 undefined-만 허용 → 첫 라이브 턴이 Zod 파싱에서 throw(200 OK 인데 화면 에러). 수정: `backend/api/main.py` `/api/chat` 에 **`response_model_exclude_none=True`** — null Optional 필드(framework·citations·uiBlocks·note) 생략 → 프론트 무수정으로 정합. 재검증: 응답 `null` 0회.

## 로컬 실행 / 데모 방법
> ⚠️ **세션 종료 시 두 서버 모두 정지함.** 라이브 챗을 다시 보려면 **매 새 세션마다 아래 2개를 직접 기동**해야 함(재생 모드는 서버 없이도 동작, 라이브 모드만 백엔드 필요). 백엔드는 `backend/.venv`(psycopg·openai 등 설치됨) + `backend/.env`(실 UPSTAGE_API_KEY·SUPABASE_DB_URL) 사용.
```
# 백엔드 (backend/) — 라이브 챗·RAG 에 필수
.venv/Scripts/python -m uvicorn api.main:app --port 8787
# 프론트 (frontend/)
npx next dev --port 3000
```
`/login` → owner/demo1234(사장님) → `/select` → 병의원 → 챗 상단 **라이브** 토글 → 질문(~14초). 프론트 `NEXT_PUBLIC_API_BASE=http://localhost:8787`·`NEXT_PUBLIC_CHAT_MODE=replay`(.env.local, gitignore). RAG A/B 는 `services/chat.ts` 의 `rag` 인자(현재 미노출; 필요 시 UI 토글 추가).

## 남은 일 (다음 세션)
1. **관리자 하차장 재편(핵심·플로우 2·3)**: `admin/pool` 을 `public.conversations` 목록으로 전환 — 생성시간·날짜·종류·소유자(가나다) 정렬 + 제목 검색 + 100 페이징 + 행 체크박스 다중선택→`tasks/new?conversationIds=` 배치. **엑셀 업로드(`admin/pool/upload`, `pool-upload.tsx`, `lib/xlsx-intake.ts`) 제거.** conversation-store(Realtime 캐시) 신설 + `getConversation` 하이드레이션(세무사 워크스페이스가 라이브 대화 원문 로드).
2. **B write-path 자동 트리거(플로우 6)**: 검수 확정(`review.finalize`) 시 accepted `line_feedback` → `ingest_feedback`(질문A+답변B+코멘트C). 이제 질문A/답변B 가 `public.conversations.payload` 에 있으므로 배선 가능해짐(직전 세션 B 잔여의 병목 해소).
3. **정산 수익% (플로우 6)**: 관리자 수익 입력 + % → pool 산정(현 `weighted_by_count` 인정건수 비례 배분 재사용). 기여의 **RAG 존속 연동**(passage 삭제 시 기여 소멸)은 별도 설계 필요.
4. 라이브 챗 pool 등록: 현재 remote 경로는 `poolService.add` 미호출(pool_candidates 는 admin-write RLS). 하차장이 conversations 를 읽으면 불필요. 통합은 1번에서.
