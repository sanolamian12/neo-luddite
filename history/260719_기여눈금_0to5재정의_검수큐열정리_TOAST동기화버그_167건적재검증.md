## 1. 시작은 잘린 글자 하나였다

검수 큐(정성 평가) 목록에서 평점 칸의 글자가 잘렸다. `평점`·`상태`·체크박스가 오른쪽
끝에 몰려 폭이 부족했다. 요청은 두 가지였다:

1. 체크박스를 가장 왼쪽 열로
2. `문장 3/5 · 법률 2/5` → `문3 / 법2` 로 축약

여기까지는 CSS 폭 조정이지만, "그럼 피드백 길이별 기여 인정 기준이 지금 뭐냐"는 질문이
따라붙으면서 눈금 자체를 다시 짜게 됐고, 검증하다가 훨씬 무거운 버그가 나왔다.

## 2. 기여 눈금 — 100자당 1(최대 10) → 6구간 0–5

### 2-1. 바꾼 것

```
0자      → 0단위   —            (빈 총평에 기여 없음)
1–100    → 1단위   100자 이하
101–200  → 2단위   200자 이하
201–500  → 3단위   500자 이하
501–1000 → 4단위   1000자 이하
1001+    → 5단위   1000자 이상
```

앞은 100자로 촘촘하고 뒤로 갈수록 넓다. 짧은 총평 사이의 100자 차이가 긴 총평 사이의
100자 차이보다 정보량 차이가 크기 때문이다.

### 2-2. 구조도 같이 고쳤다 — 눈금이 두 군데서 따로 계산되고 있었다

기존 `evalContributionUnits()` 와 `volumeLabel()` 이 **각자** `ceil(len/100)` 을 돌렸다.
그래서 라벨은 1000자에서 "1000자 이상"으로 넘어가는데 단위 상한(10)은 901자에서 이미
찍혀, **901–999자 구간에서 라벨과 단위가 어긋나 있었다.** 아무도 눈치채지 못한 건
finalize 된 건이 0건이라 단위가 화면에 나온 적이 없어서다.

→ `VOLUME_TIERS` 한 테이블에서 단위와 라벨을 함께 꺼내도록 합쳤다. 다음에 눈금을 바꿀
때 고칠 곳이 한 군데다.

`lib/audit-schema.ts:83-116`. `MAX_EVAL_CONTRIBUTION_UNITS` 10 → 5.

### 2-3. 소급 보정은 불필요했다 — 확인하고 넘어간 것

바꾸기 전에 "기존 데이터도 정리해야 하냐"를 DB 로 확인했다:

- `session_evaluations` 193건, 그중 **`finalized` 0건**
- `ledger_entries` 중 `kind='session_eval'` — **0건**

기여 단위가 DB 에 굳는 건 **최종 승인 시점의 ledger 적립뿐**이고, 목록/상세의 표시는
전부 `qualitative` 길이에서 그때그때 파생된다. 그래서 마이그레이션도 백필 스크립트도
필요 없었다. **"데이터 정리"를 요청받았을 때 먼저 물어야 할 건 "그 값이 어디에
굳어 있나"다.**

새 눈금으로 본 193건 분포 (길이 8~1814자):

| 단위 | 라벨 | 건수 |
|---|---|---|
| 1 | 100자 이하 | 1 |
| 2 | 200자 이하 | 2 |
| 3 | 500자 이하 | 10 |
| 4 | 1000자 이하 | **156 (81%)** |
| 5 | 1000자 이상 | 24 |

**4단위 한 칸에 81% 가 몰린다** — 지금 눈금은 변별력이 거의 없다. 실제 정산 배분에
쓰려면 501–1000 구간을 쪼개는 걸(예: 501–700 / 701–1000) 검토해야 한다. 이번엔
사용자 판단으로 그대로 두기로 했다. (§6 남은 것)

## 3. 진짜 버그 — Realtime 이 뺀 TOAST 컬럼

### 3-1. 증상

상세 화면에서 **[인정]을 누르는 순간 1284자 총평이 통째로 사라졌다.** 평점(3/5, 2/5)은
남고 총평만 `0자 · 기여 0` 으로 바뀌었다.

### 3-2. 원인 — WAL 에 안 실리는 unchanged-toast datum

DB 를 먼저 봤다. **행은 멀쩡했다(1284자 그대로).** 즉 화면/스토어만 비었다.

운영 DB 에 Realtime 구독을 직접 붙이고 같은 UPDATE 를 재현했더니:

```
payload.new 키 목록:
  id, scores, decision, reviewer, auditor_id, created_at,
  decided_at, decided_by, review_status, conversation_id
qualitative: 키 자체가 없음        ← 여기
```

Postgres 는 UPDATE 시 **값이 바뀌지 않은 TOAST 컬럼을 WAL 에 싣지 않는다**
(unchanged-toast datum). 긴 총평이 딱 그 대상이라, `decision` 만 바꾼 UPDATE 의
realtime echo 에는 `qualitative` 키가 아예 없다.

그런데 `makeCollectionSync` 는 payload 를 **행 전체로 보고** 도메인 객체를 통째로
갈아끼운다. `rowToEval` 의 `r.qualitative ?? ""` 가 그 빈자리를 빈 문자열로 굳혀
스토어를 덮었다.

**짧은 총평에서는 재현되지 않는다.** TOAST 임계(~2KB)를 넘는 긴 것만 걸린다. 193건 중
7건이 확실한 대상이고, 1284자(압축 1843B)짜리가 경계에서 걸렸다.

### 3-3. 고침 — 누락 키 = "변경 없음"

`lib/supabase/sync.ts` 의 `makeCollectionSync` 가 pk별로 마지막 원본 행을 들고 있다가,
realtime payload 의 빠진 키를 직전 값으로 메운다(`fillGaps`). 전체 fetch 때 캐시를 새로
깔고, DELETE 때 지운다.

- NULL 로 바뀐 경우는 payload 에 키가 `null` 로 실려 오므로 이 병합에 **안 걸린다.**
- 스토어별이 아니라 **동기화 계층**에 뒀다. `line_feedback.body` 도 같은 구조라 코멘트가
  길어지면 똑같이 터진다(현재 최대 590자로 아직 미발현). 지금 고쳐두면 그때 안 터진다.

### 3-4. 검증

운영 DB 에 실제 구독을 붙여 [인정] 클릭을 재현:

```
payload 에 qualitative 키: false
고치기 전 (payload 직접 매핑): 0자     decision=accepted
고친 후   (fillGaps 병합):    1284자   decision=accepted   => PASS
```

## 5. 대량 최종 승인 후 정합성 검증 (167건)

사용자가 검수를 진행해 167건이 최종 승인 → 배선실로 넘어갔다. 전수 대조 결과:

| 항목 | 결과 |
|---|---|
| 적재 대상 (finalized + accepted) | 167건 |
| `rag.passages` 매칭 | **167건 — 누락 0** |
| 승인 안 됐는데 실린 것 | **0건** |
| 중복 적재 | **0건** (dedupe_key = `session_eval:{evaluationId}`) |
| 임베딩 | 167/167, 전부 4096차원 |
| status | 전부 `active` |
| ledger 적립 | 167건, 누락 0 / 중복 0 / **새 눈금 불일치 0** |

```
단위 분포: 2단위 2건 · 3단위 7건 · 4단위 137건 · 5단위 21건
합계: 678단위 / 6,780 credit  (전부 auditor 앞)
```

미완: `saved` 22건 + `pending` 4건 = 26건 (최종 승인 전이라 안 실리는 게 정상).
거절 0건. KB 총계: `feedback` 354 + `session_eval` 167 + `case_seed` 8 = **529건**.

### 5-1. 검색 도달률 — 저장됐다 ≠ 검색된다

적재 확인만으로는 부족해서, 총평을 자기 질문으로 되찾을 수 있는지 표본 측정했다
(167건 중 12건 등간격 표본, top-5):

**10/12 도달 (83%), 들어온 것들의 평균 순위 2.0**

못 들어온 2건은 top-5가 **전부 `feedback` 으로 채워졌다.** 총평이 없어서가 아니라 같은
대화의 문장 코멘트들에 자리를 뺏긴 것이다. 5건 표본이던 1차 측정에서도 1건이 같은
패턴이었다 — 비율은 KB 가 커져도 유지되고 있다.

**구조적으로 예상되는 그림이다.** 한 대화에 문장 코멘트 여러 개 + 총평 1개가 같이 실리면
같은 대화 안에서 자리 경쟁이 일어난다. KB 가 더 커지면 총평이 코멘트에 밀려 검색에서
사라지는 방향이다.

## 7. 배운 것

1. **"데이터도 정리해줘"를 받으면 먼저 그 값이 어디에 굳어 있는지 찾는다.** 파생값이면
   코드만 바꾸면 끝이고, 여기선 마이그레이션을 안 써도 되는 이유가 그거였다.
2. **같은 규칙을 두 함수가 각자 계산하면 언젠가 어긋난다.** 901–999 구간의 라벨/단위
   불일치가 그랬고, 아무도 못 본 건 그 경로가 한 번도 실행된 적이 없어서였다.
3. **"화면에서 사라졌다"를 화면 버그로 보지 말 것.** DB 를 먼저 확인해 데이터가 살아
   있음을 알았고, 그 덕에 렌더링이 아니라 동기화 계층을 봤다.
4. **적재 확인과 검색 확인은 다른 일이다.** 167건이 임베딩까지 완벽히 붙어 있어도
   17%는 top-5 에 못 들어온다. RAG 는 "실렸다"가 아니라 "되찾아진다"로 검증해야 한다.
