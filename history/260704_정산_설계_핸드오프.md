# 정산(Settlement) — 설계·핸드오프 (다음 세션 콜드스타트용)

날짜: 2026-07-04 · 브랜치: `import-credigraph` · 지휘문서: [260702_마스터설계_ABC_워크스트림_분리실행.md](260702_마스터설계_ABC_워크스트림_분리실행.md)

> 이 문서 하나로 정산 세션이 콜드스타트 가능하도록 작성. 운영 흐름 6단계의 마지막 경제 로직.
> 선행: RAG write-path 완료([260704_B_writepath_검수확정_ingest배선.md](260704_B_writepath_검수확정_ingest배선.md)).

## 한 줄 요약
관리자가 **기간+금액을 직접 입력**해 정산 회차를 만들면 분배가 계산되고, "누구에게 얼마" 목록이 뜬다. 관리자는 세부화면에서 **체크박스로 일괄 [입금 처리]**, 세무사는 대시보드/마이페이지에서 **내 기여 건수·분포·비율·최종 금액·입금 상태**를 본다. 입금은 실제 계좌이체(플랫폼 밖) 후 관리자가 수동으로 상태를 넘긴다(`입금 전 → 입금 완료`).

## 사용자 확정 스펙 (2026-07-04)
1. **정산 만들기**: 관리자가 기간(YY.MM.DD ~ YY.MM.DD) + 금액을 **직접 입력**해 올린다.
2. **계산**: 자동 분배 풀이 돈다. 회차 상태 `정산 중 → 정산 완료`(비동기 OK, 짧으면 동기로 몇 초 대기 후 즉시 표시도 OK).
3. **관리자 세부화면**: 자기가 올린 정산 글에서 "누구에게 얼마" 목록을 본다.
4. **세무사 안내**: 세무사 대시보드/마이페이지에 새 정산이 **글로 열리며**, 본인의 기여 건수·분포·비율·최종 입금액이 표시된다.
5. **입금 상태 머신**: 처음 `[입금 전]`. 관리자가 **실제 입금 후** 목록 화면에서 **체크박스로 일괄 [입금 처리]** → 세무사 화면이 `[입금 완료]`로 바뀐다.
6. 이 흐름의 상당 부분은 **의도적으로 수동**(관리자 조작)이다 — 자동화하지 않는다.

## 데이터 모델 변경 (권장 최소안)
- `SettlementAllocation`(poc-schema) 에 **`paidAt?: number`** 추가(있으면 입금완료, 없으면 입금전). 별도 테이블 없이 기존 `settlement_rounds.allocations` jsonb 안에서 처리 → **Realtime 이 이미 걸려 있어 관리자 [입금 처리] 가 세무사 화면에 자동 전파**. 마이그레이션 불필요(jsonb 스키마리스).
- `markPaid` 는 read-modify-write(회차 최신 allocations 읽어 해당 auditor 항목에 paidAt 세팅 후 update). ledger 에 별도 entry 는 불필요(정산 적립은 publish 시 이미 기록됨) — 필요하면 `note` 로 입금 표기만.

## ⚠️ 분배 기준의 격리 경계 (존속 연동과의 관계 — 중요)
사용자 최종 논지는 **"지금 RAG 에 살아있는 데이터셋 기여도 분포 비율"**로 분배(메모리 [[project_operational_flow]]). 그러나:
- 현재 `preview()` 의 분배 기준 = **기간 내 ledger 인정 건수**(`weighted_by_count`). 이미 버려진 passage 의 기여분도 섞인다 → 논지와 아직 불일치.
- **핵심: 분배 기준은 `preview()` 함수 하나에 격리돼 있다.** 그래서 정산 화면/입금상태/안내 UI 는 **지금 ledger 기준으로 먼저 완성**하고, 분배 기준만 나중에 "살아있는 RAG 기여도"로 교체하면 된다. **존속 연동은 정산 착수의 blocker 가 아니다.**

**존속 연동을 "제대로" 하려면(별도 세션급) 선결 부채:**
- 분배를 `rag.passages` 의 `status='active'` 행 기준으로 바꾸려면 **passage → auditorId attribution** 필요.
- 현재 passage 는 `reviewer`(이름 문자열)·`feedback_id` 만 있고 **auditorId 없음**. 게다가 `line_feedback` 은 아직 localStorage(P2 미마이그레이션).
- 해결 경로 (택1): (a) `review.finalize`→`/api/rag/ingest` 에 **auditorId 를 실어 보내 passage 에 저장**(rag.passages 컬럼 추가) — 국소 변경, or (b) **audit-store→Supabase 마이그레이션**으로 line_feedback 서버화 후 조인 — 정공법이나 큼.
- attribution 이 서면, 존속 기여도 = `select auditor_id, count(*) from rag.passages where status='active' group by auditor_id` 로 파생 → `preview()` 가 이걸 소비. "버려지면 기여도 소멸"이 **저장이 아니라 파생**으로 자동 성립.

## 권장 실행 순서
1. **[이번 정산 세션]** poc-schema `SettlementAllocation.paidAt` 추가 → `markPaid` 서비스 → 관리자 `[roundId]` 세부화면(체크박스 일괄 입금처리) → 세무사 정산 상세 안내 뷰. **분배 기준은 현행 ledger 유지**. E2E: 만들기→발행→입금처리→세무사 화면 입금완료.
2. **[그다음/병행 세션]** attribution(권장: ingest 에 auditorId 실어 rag.passages 에 저장) → `preview()` 분배 기준을 살아있는 RAG 기여도로 교체 → 존속 연동 자동 성립(+amend/eviction retract 로 passage status 내리면 기여도도 빠짐).

## 열린 결정
- 입금 상태를 allocation `paidAt`(권장) vs 별도 payouts 테이블 — PoC 는 jsonb 내 처리 권장.
- 세무사 안내를 **대시보드 카드** vs **메일함 상세** vs 둘 다 — publish 시 이미 메일이 가므로 메일 상세 + 대시보드 배지 조합 자연스러움.
- "정산 중" 중간 상태 노출 여부(동기면 생략 가능).
- attribution 경로 (a)ingest 실어보내기 vs (b)audit-store 서버화 — (a)가 빠르고 국소적, (b)가 정공법(크로스브라우저 검수까지 해결).
