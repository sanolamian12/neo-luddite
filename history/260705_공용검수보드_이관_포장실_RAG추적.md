# 공용 검수 보드 이관 + 포장실(RAG 적재 추적)

날짜: 2026-07-05 · 브랜치: `import-credigraph` · 지휘문서: [260702_마스터설계_ABC_워크스트림_분리실행.md](260702_마스터설계_ABC_워크스트림_분리실행.md)

## 한 줄 요약
검수 코멘트/세션평가를 **localStorage(개인 사본) → Supabase 서버 공용 보드**로 이관해 "세무사 N명이 한 대화를 함께 보는 단체 채팅방"을 실현하고, 그 위에 **포장실**(RAG 로 실린 데이터셋을 대화 단위로 추적 + 연결끊기/재연결)을 신설했다. 운영 흐름 6단계의 검사실→RAG 구간이 **추적 가능한 폐루프**가 됐다.

## 배경 (사용자 개념 교정)
- 픽업하면 생기는 게 "일감" = **공용 보드(단체 채팅방)**. 대화를 복사해 가는 3개의 1:1 방이 아니라, **3명짜리 방 하나**. 같은 AI 답변 문장에 여러 세무사가 코멘트를 쌓고, 서로 확인·반박·본인 것 삭제, 관리자가 최종 불인정.
- 코멘트 모델(`LineFeedback`)은 이미 `(conversationId, segmentId)` + `reviewer` 라 보드 모양이었으나 **localStorage 라 실제 공유 안 됨**(브라우저별 개인 서랍). → 서버화가 키스톤.

## 스테이지 (4단계, 모두 라이브 검증)

### 1. DB — 마이그레이션 0007 (라이브 Tokyo 적용, 검증 10/10)
- `line_feedback`·`session_evaluations`·`rag.passages` 에 **`auditor_id`(신원=도메인 id)** 추가. `reviewer` 는 표시이름으로 분리(기존 `reviewer=current_domain_id()` RLS 와 표시이름 저장의 정합성 붕괴를 신원 컬럼 신설로 해소).
- **RLS 재설계**: 작성/수정/삭제=본인(`auditor_id=current_domain_id()`), **조회=같은 대화에 audit 있는 멤버끼리 + admin**(`feedback_member_read`/`eval_member_read`). ← "3명 방" RLS 성립.
- `session_evaluations` `unique(conversation_id)` → `unique(conversation_id, auditor_id)` (세무사별 평가).
- Realtime publication 에 두 테이블 추가.
- 적용: 재사용 스크립트 [supabase/apply_migration.py](../supabase/apply_migration.py) (`backend/.venv` psycopg + `SUPABASE_DB_URL`).

### 2. 프론트 — audit-store 컷오버 (공용 보드)
- `lib/audit-store.ts` localStorage persist → **Supabase 이중 컬렉션 동기화**(line_feedback + session_evaluations, `makeCollectionSync`). mutator write-through(낙관적). 소비처 API 보존.
- `evaluations` Record→배열, `evaluationFor(evals, convId, auditorId)` 헬퍼. `conversationStatus` 에 auditorId 인자. 11개 소비처 갱신(세무사쪽=본인, 검사실=audit 작성자).
- `addFeedback`/`setSessionEval` 에 `auditorId` 추가.

### 3. auditor_id write-path 관통
- `services/rag.ts`(IngestFeedbackItem.auditorId) → `schema.py` → `main.py` → `ingest.py`(ingest_feedback auditor_id) → `store.py`(PassageRecord.auditor_id + upsert 컬럼). passage 에 신원 실림 → **정산 존속연동 선결부채 해소**.

### 4. 포장실 (신규)
- **백엔드**: `store.list_passages(conversation_id?)` + `store.set_status(ids, status)` · `GET /api/rag/passages` · `POST /api/rag/retract`(status→retired/active). **연결끊기=삭제 아님, status 전환**이라 `match_passages`(active 만) 에서 빠지되 추적 로그 보존.
- **프론트**: `services/rag.ts` `listPassages`/`retractPassages`. `admin/packaging`(목록: 대화 단위, [연결]/[일부 끊김]/[끊김] 배지, 체크박스 일괄 연결끊기) + `admin/packaging/[conversationId]`(상세: 참여 세무사별 코멘트수 + passage 추적로그 uuid/dedupeKey/tags/status/시각 + 연결끊기/재연결). 사이드바 "포장실" (섹션 `packaging` 등록).

### 부수 — reviews 중복 버그 (0008)
- 검수 진입 이펙트(startOrGet)가 이중 실행되어 audit 당 review 2개 생성 → 재진입 시 빈 draft 를 먼저 집어 "결정 저장 안 됨"처럼 보임(실제 저장은 정상). 
- 0008 `unique(reviews.audit_id)` + 기존 중복 정리(finalized/decisions 많은 것 유지). `startOrGet` 에 inflight dedupe + DB 선확인 + on-conflict 조회.

## 데모 계정
- 2번째 세무사 **`auditor2`**(도메인 auditor2, 비번 demo1234) seed.sql + `DEMO_CREDENTIALS`(domainId/displayName). `account-store.login` 이 신원 세팅. **다인원 방 시연은 별개 브라우저 컨텍스트 2개 필요**(계정·세션 localStorage 공유).

## 검증
- **라이브 브라우저 E2E(사용자 확인)**: 일감 생성(인원 2) → auditor/auditor2 각자 픽업 → 같은 문장에 두 신원 코멘트 실시간 공존 → 관리자 검사실에서 둘 다 열람 → 인정/불인정 → 검수완료. (공용 보드 실증)
- **포장실 백엔드 HTTP E2E**: ingest(auditorId 관통) → list(provenance) → retract(status=retired, KB active 8) → reconnect(active, 9) → 정리. 통과.
- **실데이터**: finalized review 의 accepted 코멘트 4건(auditor 2 + auditor2 2)을 write-path 로 적재 → 포장실에 대화 1건·세무사 2명·코멘트 4건 노출.
- 프론트 `tsc` 0 · 신규 lint 회귀 0. 신원체인 라이브 검증(profiles.domain_id=auditor/auditor2).

## 남은 일 (다음 세션)
1. 포장실 브라우저 육안 워크스루(admin `/admin/packaging` → 상세 → 연결끊기 → KB 검색에서 빠지는지).
2. **정산 존속연동 완결**: `preview()` 분배 기준을 `rag.passages status='active'` 의 auditor_id 기여도로 교체(부채 해소됨 → 이제 파생만 하면 됨).
3. finalize 시 백엔드 미기동이면 RAG 적재가 조용히 스킵됨 → 재적재 경로(수동 재동기화 버튼) 검토.
4. amend(이의로 accept→reject) 시 자동 retract 배선.
5. 데모 전 `freeze_ms` 300000 복원 결정.
