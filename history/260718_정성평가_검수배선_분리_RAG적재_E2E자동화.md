날짜: 2026-07-18
브랜치: `import-credigraph`

## 1. 문제의식

운영자 검수실에 확정된 100건 남짓의 정성 평가(세션 총평 + 문장력/법률적 정확성 점수)가
DB 에 쌓여 있는데, **아무도 결정하지 않고 RAG 에도 전혀 흐르지 않았다.**

문장 단위 코멘트가 "이 문장이 틀렸다"를 말한다면, 총평은 "이 상담이 전체적으로 왜 부족한가"를
말한다. 후자가 실제로는 더 밀도 높은 개선 신호인데, 제품이 그걸 버리고 있었다.

### 왜 안 흘렀나 — 구조적 이유 두 개

코드를 읽고 나서야 이게 "빠뜨림"이 아니라 **막혀 있었다**는 게 드러났다.

1. **`rag.passages.source_kind` CHECK 제약에 자리가 없었다.**
   0004 가 `('feedback','kb_document','case_seed','conversation')` 4종만 허용한다.
   총평은 특정 `segment_id` 에 걸리지 않아 `feedback` 으로 위장시킬 수도 없다.
   → 넣으려 해도 DB 가 거부한다.

2. **`session_evaluations` 에 admin 쓰기 권한이 없었다.**
   `eval_owner_insert/update/delete`(0013)는 전부 `auditor_id = current_domain_id()` 소유자 한정.
   admin 은 `eval_member_read`(0007)로 **읽기만** 된다.
   → 관리자가 결정을 써도 RLS 가 막고, **RLS 가 막은 UPDATE 는 에러가 아니라 0행 갱신으로
   조용히 통과한다.** 화면은 성공한 척 하고 DB 는 그대로였을 것이다.
   (`audit-store.ts:230` deleteFeedback 이 이미 같은 함정을 겪고 주석으로 남겨둔 바 있다.)

`history/260711_세션총평_RAG유입_사각지대_증류파이프라인_설계방향.md` 에 같은 진단이
있었고 `source_kind='principle'` 을 제안했으나 미구현 상태였다.

## 2. 결정한 것

| 쟁점 | 선택 | 이유 |
|---|---|---|
| RAG 적재 범위 | **실제 적재까지** | UI 만 만들면 "핵심 개선안이 RAG 에 반영된다"는 논지가 성립 안 함 |
| 결정 저장 위치 | **`session_evaluations` 에 컬럼 추가** | 총평은 (대화, 세무사)당 1건이라 행에 직접 다는 편이 단순. `reviews.decisions`(jsonb) 합성키보다 조회가 깨끗 |
| 배선실 '규모' 기준 | **세무사 코멘트 본문 글자수 합계** | 세무사가 실제로 쓴 양을 나타냄 |
| 기여도 환산 | **100자당 1단위, 최대 10** | 문장 코멘트 1건=1단위와 같은 축. 1단위 = 기존 `CREDIT_PER_ACCEPTANCE`(10 credit) |

## 3. 구현

### 3-1. 마이그레이션 `0015`

```sql
-- 검수 결정 (총평은 (대화,세무사)당 1건이라 행에 직접)
alter table public.session_evaluations
  add column decision text,        -- accepted | rejected | null(미결정)
  add column decided_at bigint, add column decided_by text,
  add column review_status text not null default 'pending';  -- pending→saved→finalized

-- 저장·확정 건은 반드시 결정을 갖는다 (앱 가드의 DB 백스톱)
check (review_status = 'pending' or decision is not null)

-- admin 쓰기 개통 — 이게 없으면 결정이 0행 갱신으로 조용히 사라진다
create policy eval_admin_write on public.session_evaluations
  for update using (public.is_admin()) with check (public.is_admin());

-- RAG 에 자리 만들기
check (source_kind in (…기존 4종…, 'session_eval'))
```

두 게이트는 문장 단위 검수(`reviews.status draft→saved→finalized`)와 같은 리듬으로 맞췄다.

## 4. 검증

### 4-1. DB 레벨

- CHECK 제약 (savepoint 롤백): `decision='maybe'` 차단 / `review_status='done'` 차단 /
  **결정 없이 saved 차단** / 결정 있으면 통과
- admin RLS 쓰기 (PostgREST 실로그인, 브라우저와 같은 경로):
  인정·저장·확정 각 **1행 갱신**, `auditor3` 가 남의 평가 변경 시도 → **0행**(차단 정상)
- RAG 적재 (Upstage 임베딩 실호출): `source_kind='session_eval'`, 4096차원,
  `metadata.scores` 정상, 번들 형식 정상
- **두 배선실 분리**: `sourceKind=session_eval` 1건 / `feedback` 354건, 교차 오염 0

### 4-2. E2E (Playwright, 신규 도입)

`frontend/e2e/session-eval-review.spec.ts` — **3/3 통과**

## 5. 도중에 잡은 것

### 실제 UX 결함 (수정함)
첫 E2E 실패 스크린샷에서 드러났다. **1284자 총평이 [인정]/[거절] 을 스크롤 밖으로 밀어내**
[검수 저장] 만 보이는 상태였다. 총평이 1000자 넘는 게 흔하므로 구조적 문제 →
**결정 섹션을 총평 위로** 옮겼다 (평점 → 결정 → 총평).

### 배선실 제목이 대화 id 로 노출 (수정함)
E2E 스크린샷에서 배선실 (정성 평가) 항목 제목이 `live-clinic-mrls3h1s` 처럼 **대화 id**
그대로 찍혀 있었다. `getConversation()` 은 스토어를 **구독하지 않는 단순 조회**라,
대화 스토어가 나중에 하이드레이트돼도 이미 그린 행의 제목이 갱신되지 않는다.
`inspection-table.tsx:85` 가 같은 함정을 이미 주석으로 남겨 뒀는데 배선실 두 화면이
그걸 놓치고 있었다 → 두 곳 모두 `useConversationStore` 구독 추가.

### 테스트 하네스 함정 (제품 버그 아님)
- 백엔드 포트가 `NEXT_PUBLIC_API_BASE`(8787)와 다르면 적재 호출이 **연결 실패로 조용히 스킵**된다.
  비차단 설계라 최종 승인은 성공한다 — 그래서 눈치채기 어렵다.
- `CORS_ORIGINS` 에 테스트 오리진이 없으면 preflight 400 으로 막힌다.

### `next dev` 하이드레이션 — 처음 진단이 틀렸다 (해결)

처음엔 "이 환경에서 `next dev` 가 깨졌다"고 적었다. **틀렸다.** dev 서버는 멀쩡했고,
**E2E 하네스가 `127.0.0.1` 로 접속한 것**이 원인이었다.

Next 16 dev 서버는 dev 전용 엔드포인트에 대해 오리진을 검사하고, 기본값은 `localhost`
**하나뿐**이다. 허용되지 않은 오리진의 HMR 웹소켓 업그레이드는 **응답 없이 연결을 끊는다**.

curl 로 헤더를 하나씩 붙여 가며 좁힌 결과:

| Origin 헤더 | 결과 |
|---|---|
| (없음) | `101 Switching Protocols` |
| `http://localhost:3020` | `101 Switching Protocols` |
| `http://127.0.0.1:3020` | **빈 응답 = 차단** |
| `http://10.8.0.4:3020` (VPN IP) | **빈 응답 = 차단** |

`127.0.0.1` 은 `localhost` 와 같은 곳이지만 **오리진 문자열이 달라** 차단된다.
curl 은 Origin 을 안 보내서 성공했고 브라우저는 항상 보내서 실패했다 — 그래서
"서버는 되는데 브라우저만 안 된다"로 보였다.

증상이 고약한 이유: HMR 소켓이 죽으면 **React 가 하이드레이션되지 않아** 화면은 뜨는데
폼·버튼이 전부 먹통이 된다. `next build`/`next start` 는 오리진 검사가 없어 멀쩡하다.

조치: `next.config.ts` 에 `allowedDevOrigins: ["127.0.0.1", "10.8.0.4"]`.
이제 dev 서버에서도 E2E 3/3 통과(전에는 로그인조차 불가).

### 워크스페이스 루트 (부수적으로 고침)
저장소 **루트에 stray `package.json`/`package-lock.json`** 이 있었다(7/10, 루트에서
`npm i -D playwright` 한 흔적). Turbopack 이 lockfile 로 루트를 추론하다 저장소 루트를
골라 경고를 뱉고 있었다. `turbopack.root` 를 `frontend/` 로 못박아 고정.
⚠ 이 값을 바꾼 뒤에는 **`.next` 를 지워야** 한다 — 안 그러면 이전 루트로 만들어진 캐시가
남아 React Client Manifest 오류로 dev 서버가 500 을 뱉는다(실제로 겪었다).

## 6. 남은 일 — ⚠ 백엔드 재배포 필요

**프론트만 배포하면 배선실 (정성 평가) 가 잘못 보인다.**
배포된 백엔드(Oracle 도쿄)에는 아직 새 코드가 없어서:

- `POST /api/rag/ingest-session-eval` → **404** → 최종 승인은 되지만 RAG 적재가 안 된다
  (비차단이라 조용히 실패)
- `GET /api/rag/passages?sourceKind=…` → 구버전은 이 파라미터를 **무시** →
  배선실 (정성 평가) 에 문장 단위 passage 354건이 정성 평가인 척 나타난다

→ `backend/deploy/` 절차로 Seam A 를 재배포해야 완결된다.

### 그 외
- 문장 단위 배선실의 '규모'는 `line_feedback` 본문 합계라 스토어 하이드레이션에 의존한다.
  대화 수가 커지면 서버 집계로 옮기는 게 낫다.
- **로그인 직후 곧바로 다른 화면으로 이동하면** 진행 중이던 Supabase fetch 가 취소되며
  (`ERR_ABORTED`) 스토어가 "로딩 중…"에 머무는 경우를 E2E 에서 재현했다. 새로고침하면
  회복된다. 테스트는 `gotoAdmin()` 헬퍼로 우회했지만, **실제 사용자도 같은 조작으로
  겪을 수 있다** — sync 레이어에 재시도를 넣는 게 근본 대응이다.
