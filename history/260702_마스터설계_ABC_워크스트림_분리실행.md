## 1. 확정된 아키텍처 (공통 전제 — 3채팅 모두 준수)

**2-호스트 토폴로지:**
```
프론트(Next.js)  ──────────────▶  Vercel (정적/엣지, Seoul ICN 접점)
                     │  fetch
       ┌─────────────┼──────────────────────┐
       ▼                                     ▼
 Supabase (Tokyo, ap-northeast-1)     Python 컴퓨트 호스트
 = Postgres + Auth + Storage          = FastAPI(Seam A /api/chat)
 = Seam B 플랫폼 상태                    + RAG Retriever(Upstage 임베딩)
 = pgvector (RAG 벡터/그래프 저장)        + 규칙엔진(clinic_expense_engine)
```

**역할 분담:**
- **Seam A** (LLM 추론·규칙엔진·RAG) → **Python FastAPI**, 별도 컴퓨트 호스트. Supabase로 대체 불가.
- **Seam B** (감사/정산/파이프라인/계정 = 관리자→세무사→사용자 협업) → **Supabase**.
- **RAG 저장소** → Supabase **pgvector** (Python 박스 footprint ≈ 0 유지).

**컴플라이언스 선 (국내 AI 트랙):** 배포된 제품의 추론·검색·judge·임베딩 경로는 **Upstage(국산)만**. 외산(Claude 등)은 **개발 도구로만**, 런타임 제품에 배선 금지. Supabase는 DB/BaaS라 트랙 무관.

**RAG 방침 (Q2 확정):** vanilla 벡터 RAG로 시작. `Retriever` 인터페이스 + LLM 어댑터로 감싸 graph/agentic 업그레이드·LLM 교체를 국소 변경으로 열어둠. 투기적 graph/agentic 금지.

## 2. 현재 코드 상태 (각 채팅이 알아야 할 사실)

**`backend/`는 성격이 다른 4덩어리:**
| 덩어리 | 파일 | LLM | 상태 |
|---|---|---|---|
| 수집 파이프라인 | `main.py` + `collectors/` | ❌ 순수 Python | 배치 → `data/*.jsonl` |
| 분석 스크립트 | `analysis*.py` | ❌ | 배치 → `data/analysis/*` |
| 규칙엔진 | `clinic_expense_engine.py`, `creator_tax_engine.py` | ❌ | 무상태 |
| **Seam A API** | `backend/api/*` (FastAPI) | ✅ Upstage만 | **완전 무상태(DB 없음)** |

- **DB·인증·공유상태 전무.** 플랫폼 상태(Seam B)는 **프론트 Zustand + localStorage 목업**에만 존재 → 지금 구조로는 사용자 간 일감 이관 불가(= A의 존재 이유).
- **RAG 미구현.** `backend/api/pipeline.py`의 `_CASE_REF` 정규식은 임시 스텁(엔진 근거에서 사건번호만 추출). 실 그라운딩 없음(= B의 존재 이유).
- **Seam A는 준비 완료·E2E 검증됨.** `/api/chat`가 clinic 판정 경로 + follow-up 분기 실동작.

**프론트 관련(A용):**
- Zustand 스토어 ~15개: `lib/*-store.ts` (10여 개가 `persist` = localStorage).
- 서비스 계층: `frontend/services/*.ts` (pool·audit·audit-task·auditor·inquiry·ledger·mail·pipeline·review·settlement). **모두 `Promise<T>` 반환** — 백엔드 연결 시 동일 시그니처로 `fetch` 교체 설계됨(예: `services/pool.ts` 주석). **← A의 교체 지점.**

**Upstage 실측(검증완료):** base `https://api.upstage.ai/v1`(OpenAI 호환) · LLM `solar-pro3` · 임베딩 `embedding-query`/`embedding-passage` · function-calling OK. 키는 `backend/.env`(gitignore).

이 값들을 **먼저 확정**하고 각 채팅이 준수. 변경 시 이 문서를 갱신.

**3-1. Supabase — 단일 프로젝트:**
- 프로젝트 1개(`credigraph`, ref `hvnvxfakdhhbakdjkxos`), 리전 **Tokyo(ap-northeast-1)**. (생성 시점 Seoul 리전 생성 제한 → Tokyo 로 확정. 한국↔도쿄 ~30-40ms 라 §6 기준 허용. 2026-07-02 생성·라이브.)

**3-2. 환경변수 이름 규약(C가 정의, A·B가 소비):**
```

# 프론트 (frontend/.env.local)

```

**3-3. 인터페이스 경계:**
- **B → Seam A**: `Retriever.retrieve(query, k) -> list[Passage]` 를 `backend/api/`에 신설, `pipeline.py`의 `_CASE_REF` 스텁을 이 호출로 교체. **`pipeline.py`의 6단계 구조·A-3 id 규칙은 유지**(B는 ③ 근거 단계만 교체).

- **LLM 어댑터**: `backend/api/llm.py` 가 이미 provider 경계. 교체 대비해 provider-specific 하드코딩 지양.

**3-4. 호스트(C가 확정):** Python 백엔드 = Oracle Always Free 춘천(24GB) 1순위 / AWS·GCP 서울 micro(1GB, 저장소 Supabase 오프로드 강제) 대안. 기존 GPU 서버(61.109.237.114, 7/31까지)는 임시 옵션.

## 4. 워크스트림 A — Supabase 스키마·인증·Seam B 영속

**목표:** 관리자→세무사→사용자 협업이 실제로 동작하도록 플랫폼 상태를 Supabase로 이관 + 역할 기반 인증.

**범위/산출물:**
- Supabase 프로젝트 생성(Tokyo) + `public.*` 스키마(감사 task·pool·review·settlement·pipeline·ledger·mail·inquiry·account 등 스토어 대응 테이블).
- **Auth + 역할(admin / 세무사 / user)** + RLS 정책(역할별 접근 제어).
- `frontend/services/*.ts` 내부를 Zustand → Supabase 클라이언트 fetch로 교체(시그니처 불변, §3-3).
- 마이그레이션 SQL + seed 데이터(데모용).

**시작 시 읽을 것:** 본 문서 §1~3, `frontend/lib/*-store.ts`(데이터 형태), `frontend/services/*.ts`(교체 대상 시그니처), `frontend/lib/poc-schema.ts`(타입 원천).

**미결/결정 필요:** 테이블별 필드 확정(스토어 형태에서 도출), RLS 세분화 수준, 실시간(Realtime) 필요 여부(일감 이관 알림 등).

**완료 정의:** 두 브라우저(관리자/세무사)에서 한쪽 변경이 다른 쪽에 반영. 역할별 화면 접근 제어 동작.

## 5. 워크스트림 B — pgvector 벡터 RAG + Upstage Solar 자동 유지보수

**목표:** `pipeline.py`의 정규식 스텁을 실제 그라운딩으로 교체. + 세무사 코멘트 자동 인덱싱 파이프라인(전부 Upstage).

> ⚠️ **제품 논지 재정의 (2026-07-03, 메모리 `project_rag_product_thesis`):** RAG KB 는 **질문 A + Upstage 답변 B + 세무사 코멘트 C** 로 자란다. 임팩트(효과 증명)를 보이려면 **빈 KB(또는 최소 기본 지식)로 출발**해야 한다 → `all_cases.jsonl` **전량 사전 인덱싱 금지**. 실제 source 는 배포(C) 후 세무사 ~100샘플 검수 코멘트가 채운다. 따라서 B 의 무게중심은 **write path(코멘트→embedding→upsert)** 와 **on/off 임팩트 측정**에 있다.

**범위/산출물:**
- `rag.*` 스키마 + **pgvector** 테이블(passage 텍스트 + 임베딩 + 메타: 사건번호·세목·조문·결정유형).
- 인덱싱 스크립트: `backend/data/processed/all_cases.jsonl` → 패시지 분할 → `embedding-passage`로 임베딩 → pgvector 적재.
- **`Retriever` 인터페이스**(§3-3) + `pipeline.py` ③단계 교체(`_CASE_REF` → 벡터 검색). 6단계·id 규칙 유지.
- **자동 유지보수**: 신규 판례 → Solar function-calling으로 분류·태깅 → 재임베딩·업서트. (하차장→검수→포장실→RAG 루프와 연결.)

**시작 시 읽을 것:** 본 문서 §1~3, `backend/api/pipeline.py`(교체 지점 ③④), `backend/api/llm.py`(임베딩 호출 추가 위치), `backend/data/processed/all_cases.jsonl`·`data/analysis/*`(코퍼스 형태), 메모리 `reference_upstage_api`.

**미결/결정 필요:** 패시지 분할 단위(사건 전체 vs 요지 vs 문단), 임베딩 차원·거리(코사인), top-k, 하이브리드(BM25+벡터) 여부, Upstage 임베딩 쿼터.

**완료 정의:** 판정 경로 segment의 `citations`가 실제 검색된 판례로 채워짐(정규식 스텁 제거). 신규 판례 1건 자동 인덱싱 시연.

## 6. 워크스트림 C — 2-호스트 배포 토폴로지

**목표:** 호주에서 개발 → 한국 사용자 저지연으로 서비스되는 배포 구성 확정·구축.

**핵심 원칙 — "개발 위치 ≠ 서비스 지연":** 최종 사용자 지연은 **서버 리전**으로만 결정되고 개발자 위치와 무관. 호주에서 로컬 개발 후 한국 리전에 **배포**하면 됨(SSH 원격 개발은 왕복 ~150~180ms라 지양). 보너스: Upstage API 자체가 한국 → 백엔드도 한국에 두면 백엔드↔Upstage 호출도 빨라짐.

**한국/일본 리전 호스팅 옵션** (한국 사용자 기준):
| 옵션 | 리전 | 비용 | 비고 |
|---|---|---|---|
| **Oracle Cloud Always Free** ★추천 | 춘천(한국) | **영구 무료** (ARM 4vCPU/24GB) | 스펙 대비 최강·한국 소재. RAG 커져도 여유 |
| AWS EC2 | 서울 ap-northeast-2 | 12개월 무료(t2/t3.micro 750h/월) | RAM 1GB → 저장소 Supabase 오프로드 강제 |
| GCP | 서울 asia-northeast3 | $300/90일 + e2-micro 무료 | RAM 1GB → 동일 제약 |
| Naver Cloud(NCP) | 한국 | 스타트업 크레딧 | 국산 → 심사 인상·데이터 국내 |
| Fly.io / Render | 도쿄(nrt) / 싱가포르 | 소액 | 서울 없지만 도쿄 ~30~40ms면 충분 |
| 기존 GPU 서버 | 한국 | 7/31까지 | 데모 임시용, https 노출 세팅 필요 |

지연(왕복): 한국↔서울 ~5~20ms / ↔도쿄 ~30~40ms / **↔호주(시드니) ~150~180ms(비추)**. 프론트는 Vercel 엣지에 서울(ICN) 접점이 있어 어차피 빠름 → 신경 쓸 건 **Python 백엔드+Supabase를 서울(또는 도쿄)에 두는 것**뿐.

> **결론 — 추천: Oracle Cloud Always Free (춘천, 24GB, 영구무료).** 한국 소재로 저지연, RAM 24GB라 vanilla는 물론 향후 graph/agentic RAG로 커져도 덩치가 발목 잡지 않음(§4 Q2 결정 참고). micro(1GB) 티어는 저장소를 Supabase pgvector로 오프로드해야만 성립.

**범위/산출물:**
- 호스트 확정(§3-4) + Python 백엔드 배포(FastAPI, uvicorn/gunicorn, https).
- 프론트 Vercel 배포 + `NEXT_PUBLIC_API_BASE` 연결(rewrite proxy vs 직접 호출 결정).
- 환경변수 분리(§3-2) + 시크릿 관리(키 노출 방지).
- https/도메인(또는 Cloudflare Tunnel 임시), CORS 프로덕션 설정(`backend/api/main.py` 현재 dev CORS).
- 배포 절차 문서화(로컬 개발 → 배포, SSH 개발 지양).

**시작 시 읽을 것:** 본 문서 §1~3, `backend/api/main.py`(CORS·헬스), `backend/requirements-api.txt`, `backend/.env.example`, 메모리 `project_gpu_server`.

**미결/결정 필요:** 호스트 최종 선택, 도메인 유무, CI/배포 자동화 수준, 지연 실측.

**완료 정의:** 한국에서 브라우저로 배포 URL 접속 → `/api/chat` 왕복 동작. 콜드스타트/지연 허용 범위 확인.

## 7. 의존관계·실행 순서 권장

```
   [C: Supabase 프로젝트+리전 생성, 환경변수 규약 확정]  ← 가장 먼저 (A·B의 전제)
            │
     ┌──────┴──────┐
     ▼             ▼
   [A: 스키마·인증]  [B: pgvector RAG]   ← 스키마 네임스페이스 분리로 병렬 가능
     └──────┬──────┘
            ▼
   [C: 백엔드·프론트 배포 마무리]  ← A·B 산출물 위에 최종화
```
- **선행 최소 조건**: Supabase 프로젝트 1개 생성 + §3 계약 확정. 이것만 되면 A·B는 독립 병렬.
- C는 앞부분(프로젝트 생성·환경변수)을 먼저 처리하고, 배포 마무리는 A·B 진행도에 맞춰.
- 각 채팅은 작업 후 `history/`에 개별 요약 문서를 남기고, 계약(§3) 변경 시 **본 문서를 갱신**.
