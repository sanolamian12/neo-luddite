# 세션 총평(session_evaluations) → RAG 유입 사각지대 규명 + 증류 파이프라인 설계 방향

날짜: 2026-07-11 · 브랜치: `import-credigraph` · 성격: **조사 + 설계방향 합의**(구현 전, 이어가기용) · 선행: [260704_B_writepath_검수확정_ingest배선.md](260704_B_writepath_검수확정_ingest배선.md) · [260705_공용검수보드_이관_포장실_RAG추적.md](260705_공용검수보드_이관_포장실_RAG추적.md) · [260707_정산_존속연동_살아있는RAG기여도_파생.md](260707_정산_존속연동_살아있는RAG기여도_파생.md)

## 한 줄 요약
auditor 모드 실작성 테스트 중 발견: **세션 평가(문장력·법률적정확성 rating + 장문 총평 `qualitative`)는 현재 RAG에 전혀 유입되지 않는다.** `session_evaluations` 테이블에 저장 후 UI/파생상태 표시용으로만 소비되고 임베딩·인덱싱 경로가 없다. 문장별 코멘트(`line_feedback`)만 최종승인 시 RAG로 간다. 세션 총평은 **세그먼트 앵커가 없는 세션단위 지식**이라 직접 유입이 부적절 → **축적 → (Upstage)증류 → admin 게이트 → 원리 유입**의 새 2단계 파이프라인 설계가 필요하다는 데 합의. 이번 세션은 구현 없이 방향까지.

## 발견: 현재 상태 (RAG 유입 사각지대)

| auditor 산출물 | 저장 위치 | RAG 유입? |
|---|---|---|
| 문장별 코멘트 (`line_feedback`) | `line_feedback` 테이블 | ✅ 최종승인(finalize) 시 → `/api/rag/ingest` → `rag.passages` (`source_kind='feedback'`) |
| 세션평가 rating (문장력·법률정확성 1~5) | `session_evaluations.scores` (jsonb) | ❌ |
| 세션평가 장문 총평 (`qualitative`) | `session_evaluations.qualitative` | ❌ |

### 왜 안 들어가는지 (코드 경로, 검증됨)
1. **최종승인 트리거가 코멘트만 읽음** — [services/review.ts](../frontend/services/review.ts) `finalize()` (약 L217-286). L275 `ragService.ingestAcceptedFeedback(acceptedFeedback)`. `acceptedFeedback`은 `useAuditStore.getState().feedback`(=line_feedback)에서만 필터. `session_evaluations`는 이 함수에서 한 번도 안 읽음.
2. **유입 페이로드 스키마에 평가 필드 자리 없음** — [services/rag.ts](../frontend/services/rag.ts) `IngestFeedbackItem`(약 L38-52)에 `comment`/`tags`/`question`/`answerSegment`만 있고 rating·`qualitative` 슬롯 없음. `buildIngestItems`(약 L98-128)도 동일.
3. **백엔드 임베딩 텍스트도 코멘트만** — [api/rag/ingest.py](../backend/api/rag/ingest.py) `build_bundle_text`(약 L28-46): `[질문]/[AI 답변]/[세무사 코멘트]/태그`만. 엔드포인트 `POST /api/rag/ingest` → `ingest_feedback` → `source_kind='feedback'`.
4. **DB 스키마에 슬롯 자체 없음** — [supabase/migrations/0004_rag_schema.sql](../supabase/migrations/0004_rag_schema.sql) L38-39 `check (source_kind in ('feedback','kb_document','case_seed','conversation'))`. 세션평가용 source_kind 부재 → 넣으려면 CHECK 제약부터 수정.

### 용어 주의 (혼동 지점)
코드베이스에서 두 테이블이 다름:
- **`session_evaluations`** = auditor의 세션 평가(문장력·법률정확성 점수 + `qualitative` 장문). ← 이번에 논의한 "세션 총평".
- **`reviews`** = admin의 audit별 accept/reject 결정 객체(finalize 라이프사이클). ← 별개 개념.

관련 파일: [session_evaluations 스키마](../supabase/migrations/0001_public_schema.sql) (약 L116-123, `qualitative text`, `scores jsonb`), [lib/audit-schema.ts](../frontend/lib/audit-schema.ts) (`SCORE_CATEGORIES=["writing","legalAccuracy"]`), [lib/audit-store.ts](../frontend/lib/audit-store.ts) `setSessionEval`(약 L198-235)·`startEvalSync`.

## 합의된 설계 방향 (증류 파이프라인 — 구현 미착수)

세션 총평은 코멘트처럼 직접 유입하지 않는다. **원리를 증류해내는 "원재료"**로 다룬다.

```
session_evaluations (이미 쌓이는 중 = 원재료 풀)
        │   ← 별도 dump 테이블 불필요. distilled 여부 플래그만 추가하면 됨
        ▼
[증류 배치]  Upstage solar-pro3 가 N건 총평(+점수+세션맥락)을 읽어
             재사용 가능한 "원리/규칙" 후보를 구조화 출력(function-calling)
        │
        ▼
[admin 게이트]  후보 원리를 admin이 검토 → "원리 채택/기각"
             (문장단위 인정/불인정이 아님 — 총평엔 accept 앵커가 없으니 게이트를 원리 단위로 올림)
        │
        ▼
rag.passages  source_kind='principle'(신규) 로 유입 = 세그먼트 앵커 없는 세션단위 지식
```

### 왜 이 구조인지 (기존 설계와 정합)
- **직접 유입 금지**가 핵심. 총평 원문 그대로 임베딩하면 세션 특유 잡음("이 병원은…")이 RAG를 오염. 증류로 **일반화된 규칙**만 남김 → "빈 RAG로 출발, 코멘트로 자람"([[project_rag_product_thesis]]) 논지의 상위 버전.
- **admin은 문장단위 인정 불가** → 게이트를 "증류된 원리 채택 여부"로 상향. 두 게이트 철학(승인 없이 RAG 진입 불가, [[project_shared_audit_board]])은 유지.

### 미확정 결정 3가지 (다음 세션에서 확정 후 착수)
1. **증류 엔진은 Upstage 단독 (협상 불가)** — 외산 모델로 judge/증류 시 국내 AI 트랙 위반([[project_korean_track_compliance]]). solar-pro3 function-calling으로 구조화 출력([[reference_upstage_api]]).
2. **기여도/정산 귀속이 N-to-1로 꼬임 (제일 까다로움)** — 문장 코멘트는 1 auditor→1 passage라 ledger 깔끔했으나([[project_operational_flow]]의 기여=RAG존속기간, 정산=수익%×기여비율), 증류 원리는 여러 auditor 총평에서 파생. 하나의 원리 passage 기여도를 원천 총평 auditor들에게 어떻게 분배(균등? 인용 가중?)할지 정해야 [260707 살아있는 기여도](260707_정산_존속연동_살아있는RAG기여도_파생.md) 모델이 안 깨짐. → **착수 전 반드시 결정**.
3. **증류 트리거** — 데모 관점 admin 수동 "원리 추출" 버튼 권장(임계치 자동보다 통제·시연 가능). 배치 대상 = `session_evaluations` 중 `distilled=false`.

## 착수 시 예상 산출물 (아직 아무것도 안 만듦)
- 마이그레이션: `session_evaluations.distilled` 플래그 + `rag.passages` CHECK 제약에 `'principle'` 추가(+ 원천 attribution 컬럼/조인 테이블).
- 백엔드: 증류 엔드포인트(solar-pro3 호출 → 원리 후보), 원리 유입 경로(`source_kind='principle'`), admin 게이트용 조회/승인 API.
- 프론트: admin "원리 추출/검토/채택" UI.
- 정산: N-to-1 기여도 분배 로직 반영.
