## 0. 이 세션의 한 줄 요약

> **쓰기 경로가 브라우저 → Supabase 직접 insert 라, 규칙을 UI 에만 걸면 규칙이 아니다.**
> 확정·제출·분류·중복 — 네 규칙을 전부 **UI · 스토어 · DB(RLS/CHECK/유니크)** 3중으로 내렸다.

이 코드베이스에는 Next.js API route 도 server action 도 없다(`find app -name route.ts` → 0건).
모든 도메인 쓰기는 zustand 스토어의 write-through 로 Supabase 에 직접 꽂힌다. 즉 **클라이언트
가드는 "안내"일 뿐 강제가 아니다.** 다른 탭·경쟁 삽입·구버전 프론트·직접 호출이면 그냥 뚫린다.
그래서 이번 수정은 전부 DB 를 최종 방어선으로 두고, UI 가드는 "왕복 없이 즉시 알려주는" 역할만 맡긴다.

## 1. 결함 ① 확정된 대화에 코멘트가 계속 들어왔다 (마이그레이션 0012)

**증상.** admin 이 [최종 승인]으로 `확정` 상태를 만든 대화에, 세무사 계정이 들어와 코멘트를 계속 추가할 수 있었다.

**왜 치명적인가.** 확정 뒤에는 `services/review.ts` 의 `setDecision`/`amendDecision` 이 throw 한다
(= 인정도 거절도 불가). 따라서 확정 후에 들어온 코멘트는 **영영 '보류'로 남아** 검수 큐를 오염시킨다.

**근본 원인.** 세무사 쪽에 확정 여부를 보는 눈이 아예 없었다. `line-feedback-panel.tsx` 는
`audit.status` 도 `review.status` 도 참조하지 않았다.

**수정.**
- `isConversationFinalized(reviews, audits, conversationId)` — 검수는 **대화 단위**라
  형제 audit 중 하나라도 `finalized` 면 그 대화 전체를 잠근 것으로 본다(`review-lookup.ts`).
- DB: `conversation_finalized(text)` SECURITY DEFINER 함수 + `line_feedback` 쓰기 RLS.
- **같은 구멍의 이웃도 막음**: 확정된 대화에 세무사가 **제출**까지 할 수 있었다 →
  관리자 검수 큐에 "결정 불가능한" 검수 건이 새로 생긴다. `SubmitPanel` 에도 잠금.

## 2. 결함 ② 분류(태그) 없이 코멘트가 저장됐다 (0012)

**증상.** 「법적 해석 오류 / 문법적 오류 / 제안사항」 중 아무것도 고르지 않고 코멘트 저장 가능.

**왜 치명적인가.** 무태그 코멘트는 **RAG 적재 시 어느 갈래로 넣을지 판단할 근거가 없다.**
(운영 흐름 6단계의 마지막 = 최종승인 → RAG 적재. 거기서 AI 가 헤맨다.)

**근본 원인.** 4개 레이어가 **전부** 빈 배열을 허용했다.

| 레이어 | 그때 | 지금 |
|---|---|---|
| zod | `tags: z.array(feedbackTagSchema).default([])` | `.min(1)` |
| UI | `disabled={!body.trim()}` — 태그 미검사 | 태그 0개면 저장 불가 + 사유 표시 |
| 스토어 | 검증 없이 insert | `addFeedback` 이 사유 문자열 반환 |
| DB | `tags text[] not null default '{}'` | `CHECK (cardinality(tags) >= 1 and tags <@ ...)` |

## 3. 결함 ③ 똑같은 코멘트가 반복 저장됐다 (0012)

**규칙.** 한 대화의 한 문장에 대해, **작성자 불문** 정규화 문자열이 같으면 두 번째부터 거절.
정규화 = 앞뒤/중복 공백 축약 + 소문자화. 프론트 `feedbackDedupeKey()` 와 DB 인덱스 표현식
`lower(regexp_replace(btrim(body), '\s+', ' ', 'g'))` 이 **같은 규칙**이어야 한다(어긋나면 UI 는
통과시키는데 DB 가 거절하는 상황이 난다).

- UI: 입력 중 실시간 판정 → 버튼 잠금 + "이 문장에 동일한 코멘트가 이미 있습니다 (OO 님 작성)"
- DB: `unique (conversation_id, segment_id, 정규화 body)` — 공용 보드라 **남이 쓴 같은 말도 중복**이다.

## 4. 결함 ④ 제출된 일감을 계속 고칠 수 있었다 (마이그레이션 0013)

**증상.** `제출됨`·`검수저장` 상태의 일감에서 세무사가 코멘트·세션평가를 계속 추가·수정·삭제할 수 있었다.
제출 화면은 "제출 후에는 수정할 수 없습니다"라고 **이미 약속하고 있었지만 강제하는 곳이 없었다.**

**왜 치명적인가.** 관리자가 검수 화면에서 본 것과 결정 대상이 어긋난다(이미 인정/거절한 코멘트의
**본문이 바뀌는 것**까지 가능). 검수 큐에 결정 없는 '보류'가 뒤늦게 끼어든다.

**잠금 단위가 핵심 — 평가자별이다.**
공용 보드라 한 대화를 여럿이 본다. 내가 제출했다고 **아직 draft 인 공동 평가자까지 막으면 안 된다.**
→ `my_audit_submitted(conversation_id)` = "내 audit 이 draft/cancelled 가 아닌가". `cancelled`(포기)는 제출로 치지 않는다.

| 잠금 | 함수 | 범위 | 트리거 |
|---|---|---|---|
| 확정 잠금 | `conversation_finalized()` | **대화 전체** (모든 평가자) | 형제 audit 중 하나라도 finalized |
| 제출 잠금 | `my_audit_submitted()` | **평가자별** (나만) | 내 audit 이 draft/cancelled 아님 |

**세션 평가도 함께 잠갔다.** 코멘트만 막으면 [평가] 탭에서 점수·정성평가를 제출 뒤에 계속 고칠 수 있어
같은 구멍이 그대로 남는다. 잠긴 상태에선 편집기 대신 제출한 평가를 읽기 전용으로 보여 준다.

**설계 확정(사용자 확인, 2026-07-14):**
> **세무사가 제출 후 자기 코멘트를 고쳐서 재검수받는 흐름은 만들지 않는다.**
> 이의제기는 **운영자가 인정/거절을 재결정**(`review.decisions`)할 뿐이고 코멘트 본문은 불변이다.
> → 제출 잠금에 예외(이의 구간 편집 허용)를 열 필요가 없다. 나중에 이 잠금을 완화하려는 시도가
> 있으면 이 문단을 근거로 막을 것.

## 5. RLS 를 쪼갠 이유 (0012·0013 공통 — 재사용할 패턴)

0007 은 `feedback_owner_write` / `eval_owner_write` 를 **`for all`** 로 만들었다. `for all` 은
**SELECT 까지 묶인다** → 거기에 잠금 조건을 얹으면 **본인 코멘트 조회까지 막힌다**(제출하면 내가 쓴 걸
못 보게 된다). 그래서 행위별로 쪼갰다:

```
feedback_owner_read    for select  using (본인)
feedback_owner_insert  for insert  with check (본인 and 미확정 and 미제출)
feedback_owner_update  for update  using/with check (동일)
feedback_owner_delete  for delete  using (동일)
-- 조회는 0007 의 feedback_member_read(같은 대화 멤버 + admin)가 그대로 유지
```

`conversation_finalized` / `my_audit_submitted` 는 **SECURITY DEFINER** 여야 한다. 세무사 세션에서
`reviews`/`audits` 의 RLS 에 걸려 확정·제출 여부를 잘못(false 로) 판정하면 잠금이 통째로 무력해진다.

**함정 하나 — RLS 가 막은 DELETE 는 에러가 아니다.** USING 절이 행을 걸러낼 뿐이라 "0행 삭제"로
조용히 성공한다. 그래서 `deleteFeedback` 은 `.select("id")` 로 **실제 지워진 행을 확인**해야 한다.
(INSERT 는 WITH CHECK 위반이 42501 에러로 뜬다 → 이건 잡힌다.)

## 6. 낙관적 갱신 롤백 (부수 수정이지만 중요)

기존 `addFeedback`/`deleteFeedback` 은 **DB 쓰기가 실패해도 낙관적 갱신을 되돌리지 않고** `console.error`
만 찍었다. → 저장되지 않은 코멘트가 **화면에만 남는다.** DB 제약을 새로 거는 순간 이 버그가 바로
표면화된다(구버전 프론트 + 신버전 DB = 조용한 유실). 이제 실패 시 롤백 + 사유 문자열 반환:

```ts
addFeedback: (...) => Promise<string | null>   // null = 성공, 문자열 = 사용자에게 보여줄 사유
// 23505 → "같은 문장에 동일한 코멘트가 이미 있습니다."
// 23514 → "분류를 최소 1개 선택하세요."
// 42501 → "더 이상 코멘트를 남길 수 없습니다 — 이미 제출했거나 검수가 확정된 대화입니다."
```

> **배포 순서 교훈:** 마이그레이션을 먼저 올리고 프론트를 늦게 올리면, 그 사이 구버전 프론트 사용자는
> "저장된 줄 알았는데 사라지는" 경험을 한다. 다음엔 프론트 롤백 처리를 **먼저** 배포할 것.

## 7. TS 함정 — zustand 초기화 함수 안에서 스토어 자기 참조

`addFeedback` 안에서 중복 검사를 하려고 `useAuditStore.getState().feedback` 을 참조했더니
**스토어 타입 전체가 `any` 로 무너지고** 소비처 전역에 implicit-any 71개가 쏟아졌다.

```
TS7022: 'useAuditStore' implicitly has type 'any' because it does not have a type annotation
        and is referenced directly or indirectly in its own initializer.
TS7023: 'addFeedback' implicitly has return type 'any' ...
```

**원인.** 반환값이 있는 액션이 초기화 함수 안에서 스토어 자신을 참조하면 반환 타입이 순환한다.
(기존 `setSessionEval` 도 자기 참조를 했지만 `void` 라 순환이 안 생겨 우연히 통과하고 있었다.)

**해법.** `create<AuditState>()((set, get) => ...)` 의 **`get()`** 을 쓴다. `get()` 은 제네릭으로
`AuditState` 가 이미 박혀 있어 순환이 없다. → 스토어 액션에서 현재 상태가 필요하면 **항상 `get()`**.

## 8. 데이터 정리 (운영 DB, 사용자 승인 후 실행)

마이그레이션 전 실측 → 유니크 인덱스는 기존 중복이 남아 있으면 **생성 자체가 실패**하므로 선정리 필수.

| 항목 | 값 |
|---|---|
| 코멘트 총계 | 584 → **367** |
| 무태그 행 | 217 (37%) → 전량 삭제 |
| 정확중복 행 | 37 → **전부 무태그 행과 겹침** (태그 없이 연타로 남긴 것들) |
| 무태그 중 RAG 적재분 | **0** |
| 무태그에 붙은 관리자 결정 | **0** |
| 확정 대화 소속 중복 | **0** |

→ 고아 `rag.passages`·고아 `decisions` 가 생기지 않음을 **확인한 뒤** 지웠다. 실제 삭제는 217건뿐.

**남은 부채 (미해결):** 삭제한 217건 중 **17건은 이미 확정된 대화 소속**이라 ledger 기여도가 이미

기여도를 가진 세무사가 있다.** 정산 전에 처리 방침 결정 필요.

## 9. 결함 ⑤ 사이드바 [완료] 배지가 98, 실제 완료는 40

**원인.** `countUnseenResults` 의 모집단에 audit 상태 필터가 없었다. 완료 화면(`ResultsTable`)은
내 audit 중 `submitted`/`reviewed`/`finalized` 만 세는데(40), 배지는 **`draft` 까지 훑는다**.
공용 보드라, 내가 아직 작성 중인 대화라도 **공동 평가자가 먼저 제출해 형제 audit 에 review 가 붙으면**
`reviewForAudit` 의 형제 폴백에 걸려 "결과가 열렸는데 안 본 건"으로 잡혔다 → 40 + draft 58 = 98.

**수정.** 배지 모집단을 완료 화면과 동일하게(제출 이상) 맞춤. 운영 DB 실측 검증:

| 계정 | 내 일감 | draft | 완료 | 옛 배지 | 새 배지 |
|---|---|---|---|---|---|
| auditor | 100 | 60 | 40 | 98 | **38** |
| auditor2 | 100 | 70 | 30 | 99 | **29** |
| auditor3 | 100 | 67 | 33 | 99 | **32** |

> **일반화된 교훈:** 배지 카운트와 그 배지가 가리키는 목록 화면은 **같은 필터를 공유**해야 한다.
> 공용 보드에서 "형제 audit 폴백"은 편리하지만, 모집단을 좁히지 않으면 남의 진행 상황이 내 숫자로 샌다.

## 11. 검증 방법 (다음에도 이렇게 할 것)

**RLS 는 superuser 로 검증하면 안 된다.** psycopg 로 붙으면 RLS 를 우회하므로 전부 통과한다.
세무사 세션을 흉내내야 진짜 검증이다:

```python
cur.execute("set local role authenticated")
cur.execute("select set_config('request.jwt.claims', %s, true)",
            (json.dumps({"sub": auth_uid, "role": "authenticated"}),))
# → 이제 current_domain_id() 가 그 세무사를 가리킨다
```

실제 검증 결과(auditor2, `reviewed` 상태 일감):
- 코멘트 insert → **거절**(InsufficientPrivilege) · update/delete → **0행** · 세션평가 update → **0행**
- 조회는 정상(2건) — 제출해도 볼 수는 있어야 한다
- **draft 인 공동 평가자의 insert 는 허용됨** ← 평가자별 잠금이 과잉 차단하지 않는지 확인

**브라우저 검증은 Playwright 가 이미 설치돼 있다**(`node_modules/.bin`, v1.61.1). 스크립트를
`frontend/` 안에 두고 실행해야 모듈이 해석된다. 로그인은 `#username`/`#password` + `button[type=submit]`
(데모: auditor / demo1234).
