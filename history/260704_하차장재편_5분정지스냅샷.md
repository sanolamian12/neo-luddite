# 하차장 재편 — 라이브 대화의 "5분 정지 스냅샷" + 엑셀 폐지

날짜: 2026-07-04 · 브랜치: `import-credigraph` · 지휘문서: [260702_마스터설계_ABC_워크스트림_분리실행.md](260702_마스터설계_ABC_워크스트림_분리실행.md)

## 한 줄 요약
관리자 하차장을 엑셀 업로드에서 **사장님이 만든 라이브 채팅창의 "5분 정지 스냅샷" 목록**으로 재편했다. 채팅방은 얼리지 않는다(사장님은 계속 대화 가능) — pg_cron 이 채팅방 생성 5분 뒤 그 시점의 대화를 **사진 한 장**으로 복사(`snapshot_payload`)하고, 하차장·일감·RAG 는 이 정지 사본만 소비한다. 정지 데이터라 라이브 진행 상황과 싱크가 어긋나지 않는다. 엑셀 체인·구 `pool_candidates` 전량 폐지.

## 핵심 정책 (사용자 확정)
- **문제의식**: 라이브 대화를 즉시 하차장에 적재하면, 채팅은 계속 흐르는데 일감은 정지된 데이터를 전제해 싱크가 깨진다.
- **해결**: 채팅방 생성 후 **5분 시점에 사진을 찍는다**. 그 사진을 하차장에 들고온다. 5분 넘긴 이후 턴은 사진에 담기지 않는다(대부분 대화가 5분 내 종료된다는 가정).
- **중요**: 채팅방 자체는 얼리지 않는다 — 사장님은 5분 후에도 계속 대화 가능. 하차장/일감/RAG 가 보는 것만 정지 사본이다.
- **동결시간**: `public.app_config.freeze_ms`(서버 단일 소스). 기본 300000(5분), 데모/E2E 시 낮춰 단축. **현재 데모용 60000(1분)로 설정됨.**

## 산출물

**DB (마이그레이션 0006 — 라이브 적용):**
- `supabase/migrations/0006_conversation_snapshots.sql`
  - `public.conversations` 컬럼 확장: `snapshot_at`(사진 시각)·`snapshot_payload`(정지 사본 jsonb)·`excluded_at`(관리자 제외). 별도 테이블 대신 컬럼 확장 → 라이브↔스냅샷 1:1 자연 보장 + 0005 Realtime 재사용.
  - `public.app_config(key,value)` + `freeze_ms` 시드. RLS admin-only.
  - `snapshot_settled_conversations()`: `snapshot_at is null AND age ≥ freeze_ms` 인 라이브 대화의 현재 payload 를 복사(라이브 payload 는 불변경). security definer.
  - **불변 트리거** `enforce_snapshot_immutable`: 사진은 한 번 찍히면 변경 불가(사장님 라이브 upsert 가 덮으려 해도 조용히 기존값 복원 — 라이브 챗은 절대 안 막음).

## 검증
- **DB 라이브(Tokyo credigraph)**: 0006 13문 적용 + `schema_migrations 0006` 기록. 컬럼·app_config·pg_cron·함수·트리거·cron job(active, `* * * * *`) 확인.
- **cron 라이브 동작**: 기존 라이브 대화 1건(age≈35분)을 스케줄러가 자동 스냅샷 → `snapshot_at` 세팅, `cron.job_run_details` = **succeeded, "1 row"**. 5분 정지 파이프라인 라이브 증명.
- **데이터 계층 E2E**: ① `select *` 계약 — 새 컬럼 전부 실림, eligible 1건 ② 불변 트리거 — snapshot_payload 덮어쓰기 시도 보존 **PASS** ③ setExcluded — 제외/복원이 eligibility 정확히 토글 ④ freeze_ms→60000.
- **정적**: `tsc --noEmit` 0 · `next build` 전 라우트 컴파일 성공(`/admin/pool/upload` 소멸 확인) · 신규 lint 회귀 0(남은 lint 에러는 레포 baseline, 미변경 라인).
- **잔여**: 브라우저 육안 워크스루(라이브챗→1분 뒤 스냅샷→하차장 노출→선택→task). 두 서버 기동 필요(데모 시).

1. **브라우저 워크스루**: freeze_ms=60000 상태에서 라이브 챗 생성→1분 뒤 하차장 자동 노출(Realtime)→다중선택→Task 배치 육안 확인.
2. **B write-path 자동 트리거(플로우 6)**: 검수 확정 시 accepted `line_feedback` → `ingest_feedback`. 질문A/답변B 원문이 `snapshot_payload` 에 정지 상태로 있으므로 배선 가능.

4. 데모 전 `freeze_ms` 를 다시 300000(5분)으로 되돌릴지 결정(현재 1분).
