# [검수확정]이 draft 형제 audit 을 안 닫아, 확정된 건이 세무사 '진행중'에 영영 남던 문제

작성일: 2026-07-14
상태: **배포 완료** (`main` = `4e24c04`). 마이그레이션 0014 운영 DB 적용됨.
계기: auditor2 계정으로 로그인해 `/audit/work`(진행중 71건)를 보니, admin 이 이미 **검수확정**한 건들이 그대로 목록에 떠 있었다.

---

## 0. 이 세션의 한 줄 요약

> **[검수확정]은 그 대화의 리뷰·기여를 "참여자 전원"에 대해 종료하는 사건이다.**
> 그런데 코드는 *이미 제출한* 평가자에게만 그 종료를 전파했다. 아직 제출하지 않은 평가자의
> audit 은 `draft` 로 방치돼, 손댈 수도 없는 **유령 작업**으로 진행중 목록에 영원히 남았다.

실측해 보니 이게 예외적 잔재가 아니었다. **draft 199건 중 197건이 이미 확정된 대화 소속**이었고
(reviews 100건 전부 `finalized`), 시스템 전체에 살아 있는 진짜 작업은 **2건**뿐이었다.
세무사가 보던 "진행중 71건"은 사실상 전부 허수였다.

---

## 1. 증상

- auditor2 의 진행중 목록 71건 중 대부분이 admin 이 최종 승인을 끝낸 대화였다.
- 그 항목들은 "총 피드백 3 / 나의 피드백 0 · 시작전" 형태 — 즉 **다른 세무사가 코멘트를 달고
  제출했고, admin 이 확정까지 마친 대화**인데, auditor2 의 몫만 남아 있었다.
- 열어 봐야 할 수 있는 일이 없다. 확정 대화는 0012 RLS(`conversation_finalized`)가 코멘트
  쓰기를 막는다. **작업 불가능한데 목록만 차지하는 항목** = 큐 오염.

## 2. 근본 원인 — `siblingAudits()` 가 draft 를 형제로 안 봤다

검수는 **대화 단위**다. 한 대화를 여러 평가자가 평가해도 review 는 대표 audit 하나에만 붙고,
상태 전이를 형제 audit 에 전파해야 한다. 그 전파 대상을 고르는 함수가 draft 를 빼고 있었다.

```ts
// services/review.ts (수정 전)
function siblingAudits(conversationId: string) {
  return audits.filter(
    (a) => a.conversationId === conversationId &&
      (a.status === "submitted" || a.status === "reviewed" || a.status === "finalized"),
  );  // ← draft 가 없다
}
```

`finalize()` 는 이 목록만 `finalized` 로 바꾼다 → **아직 제출하지 않은 평가자의 draft 는
그대로 남는다.** 진행중 목록의 필터는 `status === "draft"` 하나뿐이라(`work-table.tsx`),
그 유령들이 전부 통과했다.

원래 이 필터는 `save()` 를 위한 것이었다(저장 게이트는 `submitted` → `reviewed` 만 승격).
`finalize()` 가 같은 함수를 재사용하면서 의미가 어긋났다 — **저장은 "결과를 여는" 사건이지만,
확정은 "작업을 닫는" 사건**이라 모집단이 다르다.

부수 피해 하나 더: ledger 적립 루프도 같은 `siblings` 를 돌았다. 그래서 **draft 인 채로 코멘트만
쓴 공동 평가자는 자기 코멘트가 인정돼 RAG 에 적재됐는데도 기여 적립을 못 받았다**
(RAG 적재는 `conversationId` 기준이라 작성자의 audit 상태와 무관하게 들어갔다).

## 3. 수정

### ① 확정이 draft 형제까지 닫는다 (`services/review.ts`)

`siblingAudits()` 는 `cancelled` 만 제외하고 draft 를 포함한다. `finalize()` 는 형제를 둘로 가른다.

| 기여(코멘트 또는 세션평가) | 결과 상태 | 근거 |
|---|---|---|
| 있음 | `finalized` | 그 코멘트도 관리자의 인정/거절 대상이었다 → 완료 목록에서 결과를 볼 자격이 있고, **ledger 적립 대상**(위 부수 피해도 함께 해소) |
| 없음 | `cancelled` | 보여 줄 결과가 없다 → 진행중·완료 어디에도 뜨지 않는다 |

`save()` 는 그대로 `submitted` 만 승격한다(저장 단계는 아직 이의 구간 — 세무사는 계속 작성 가능).
**닫는 건 오직 확정 게이트뿐**이라는 두 게이트 모델이 여기서 명확해졌다.

### ② 방치분 소급 정리 (마이그레이션 `0014_finalize_closes_drafts.sql`)

코드 수정은 앞으로만 유효하다. 이미 쌓인 197건을 같은 규칙으로 정리했다.

| | 적용 전 | 적용 후 |
|---|---|---|
| draft | 199 | **2** |
| finalized | 103 | 159 (+56) |
| cancelled | 0 | 141 |

- 기여 있음 56건(auditor 51 · auditor3 5) → `finalized`
- 기여 없음 141건(auditor 9 · **auditor2 70** · auditor3 62) → `cancelled`
- 검증: `select count(*) from audits a where a.status='draft' and conversation_finalized(a.conversation_id)` → **0**
- **ledger 는 소급 적립하지 않았다** — append-only 통장이라 과거를 정정하지 않는다(0012 와 동일 원칙).

### ③ 프론트에도 같은 규칙 한 겹 (`isOpenDraft()`)

`review-lookup.ts` 에 `isOpenDraft(reviews, audits, audit)` = `draft && !확정된 대화`.
`work-table.tsx`(목록)와 `sidebar-badges.ts`(배지)가 같은 모집단을 쓴다.

DB 를 고쳤는데 왜 프론트 가드도 두나 — **확정 이후에 그 대화가 낀 Task 를 픽업하면 신규 draft 가
또 생긴다**(`audit-task.ts::pickup` 은 task 의 모든 conversation 에 draft 를 만든다). 이 경로는
finalize 가 이미 지나간 뒤라 코드 ①로는 못 막는다.
0011 RLS 덕에 세무사도 같은 대화 멤버의 review 를 읽을 수 있어 클라이언트에서 확정 판정이 가능하다.
(review 하이드레이션 전에는 판정 불가 → 로딩 게이트에 `reviewHydrated` 추가, 깜빡임 방지.)

## 4. 남은 것 / 배운 것

- **픽업 자체를 막지는 않았다.** 확정된 대화가 낀 Task 를 픽업하면 DB 엔 여전히 쓸모없는 draft 행이
  생긴다(화면엔 안 뜬다). RLS 로 insert 를 막으면 batch insert 전체가 실패해 픽업 자체가 깨지므로
  일부러 두지 않았다. 필요하면 `pickup()` 에서 `conversation_finalized` RPC 로 걸러내면 된다.
- **교훈: 상태 전이 함수가 "모집단"을 공유하면 의미가 어긋난다.** `save`(여는 사건)와
  `finalize`(닫는 사건)는 대상 집합이 다르다. 헬퍼 하나를 둘이 나눠 쓰면서 조용히 어긋났고,
  그 결과가 유령 197건이었다.
- **교훈: 목록 필터는 상태 하나만 보면 안 된다.** `status === 'draft'` 는 "내가 안 끝냈다"만 말하지
  "아직 할 수 있다"를 말하지 않는다. 진행중의 정의는 `draft && 대화가 안 닫힘` 이어야 했다.
