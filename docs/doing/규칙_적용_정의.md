## 1. 결정: Citation + Validation 결합

| 옵션 | 측정 방식 | 결정론성 | 트랙 정합 | 채택 |
|---|---|---|---|---|
| **1. 직접 인용 (Citation)** | Solar Pro 응답의 `cited_rule_ids` 추출 | 명확 (function calling) | OK | ✅ Primary |
| 2. 의미 참조 (Semantic) | 임베딩 유사도 / LLM judge로 사후 분석 | 모호, 위양성 발생 | ⚠️ LLM judge 시 트랙 위반 | ❌ 배제 |
| **3. 후처리 검증 (Validation)** | 규칙 validator로 자동판정 결과 검증 | 결정론적 | OK | ✅ Secondary |

**의미 참조 배제 이유**:
- LLM judge는 외산 모델 사용 시 트랙 위반, 국산 모델 사용해도 위양성 모호성 잔존
- Solar Embedding 기반 유사도도 임계값 임의성 문제
- Citation + Validation 결합만으로 차별점 가시화 충분 — 모호 신호를 굳이 끌어들이지 않음

## 2. "규칙 적용됨" 형식적 정의

규칙 R이 케이스 C에 적용된 경우 — 다음 두 경로 중 **하나 이상 충족**:

| Mode | 조건 |
|---|---|
| **(a) citation** | Consensus Engine이 R의 텍스트를 Solar Pro 컨텍스트에 주입했고, 응답의 `cited_rule_ids`에 R.rule_id 포함 |
| **(b) validation** | 자동판정 결과를 R의 validator에 입력했을 때 `Pass` 또는 `Fail` 판정 (N/A 제외) |

→ 둘 중 하나라도 충족 시 `rule_applied{rule_id, case_id, mode}` 이벤트 1건 기록. mode = `citation | validation | both`.

**파생 변수**:
- `n_applications(R)` = R에 대해 mode 무관 unique case 수 — Reusability 곱셈 변수
- `has_applied_rule(C)` = (C에 적용된 규칙 수 ≥ 1) — KPI 4-A 분자

## 3. 규칙 데이터 모델 v0.1

```yaml
rule_id: R-042
category: deduction_eligibility | calculation | exception | procedure
title: "의료비 공제 한도 — 본인·부양가족 합산"
natural_language: |
  본인 및 부양가족의 의료비는 합산 후 총급여액의 3%를 초과하는
  부분에 대해서만 공제 대상이 된다. (소득세법 §52 ②)
validator:                            # optional. 없으면 citation 모드로만 적용 가능
  preconditions:
    - field: deduction.type
      op: equals
      value: medical
    - field: taxpayer.total_income_krw
      op: exists
  assertion:
    - field: deduction.eligible_amount_krw
      op: equals
      value: "max(0, sum(medical_expense) - 0.03 * total_income_krw)"
source_diffs: [review-2026-06-12-008, review-2026-06-19-021]
author_expert_id: expert-04
approved_at: 2026-06-20T11:00:00Z
approved_by_admin: admin-01
```

**필수 vs 선택**:
- 필수: `rule_id`, `category`, `title`, `natural_language`, `source_diffs`, `author_expert_id`, `approved_at`, `approved_by_admin`
- 선택: `validator` — 코드 검증 가능한 규칙만 추가. 자유형 규칙은 citation-only로 작동

**source_diffs 필수 이유**: Knowledge Asset Extractor가 어떤 검수 diff에서 이 규칙을 추출했는지 추적 — KPI 5-A 원장 추적률 100% 보장에 직결. 직접 등록 규칙도 `source_diffs: []` (빈 배열)로 명시.

## 4. Validator DSL v0.1

자유형 Python 실행은 보안·검증 비용이 크므로 선언적 DSL로 한정.

### FieldCheck 스키마

```yaml
FieldCheck:
  field: "dot.notation.path"      # 자동판정 결과 JSON 경로
  op: equals | not_equals | in | not_in | range | regex_match | exists
  value: any                       # op에 따라 의미 다름
```

### 지원 연산자

| op | value 형식 | 의미 |
|---|---|---|
| equals | scalar | 정확 일치 |
| not_equals | scalar | 불일치 |
| in | array | 멤버십 |
| not_in | array | 비멤버십 |
| range | [min, max] (null 허용) | 범위 (한쪽 null 가능) |
| regex_match | string | 정규표현식 매치 |
| exists | (없음) | 필드 존재 여부 |

### 평가 의미

| 상태 | 조건 |
|---|---|
| **N/A** | preconditions 미충족 → Reusability 카운트 제외 |
| **Pass** | preconditions 충족 + assertion 모두 충족 → 적용 인정 |
| **Fail** | preconditions 충족 + assertion 일부 미충족 → 적용 인정 (규칙이 자동판정 오류 검출) |

Pass·Fail 모두 적용으로 인정하는 이유: Fail은 "규칙이 자동판정의 오류를 잡았다"는 강한 신호 — ErrorPrevention 이벤트의 단서.

### v0.1 한계

- 산술식 평가는 `value: "max(0, sum(medical_expense) - 0.03 * total_income_krw)"` 같은 표현을 별도 평가기 필요. v0.1에서는 **상수·필드 참조만 지원**하고, 산술 비교는 본선 후 확장.
- 다중 필드 cross-validation (예: field A < field B)은 v0.1 미지원

## 5. Solar Pro Function Calling으로 Citation 강제

Citation 측정을 결정론적으로 만들기 위해 structured output 사용.

```python
tools = [{
  "type": "function",
  "function": {
    "name": "submit_consensus_judgment",
    "description": "본 케이스에 대한 합의 판정 제출",
    "parameters": {
      "type": "object",
      "required": ["decision", "reasoning", "cited_rule_ids", "confidence"],
      "properties": {
        "decision": {"type": "string"},
        "reasoning": {"type": "string"},
        "cited_rule_ids": {
          "type": "array",
          "items": {"type": "string"},
          "description": "본 판정에 참고한 규칙 ID 목록. 없으면 빈 배열."
        },
        "confidence": {"type": "number", "minimum": 0, "maximum": 1}
      }
    }
  }
}]
```

**system prompt에 명시**: "제공된 규칙을 참고했다면 반드시 `cited_rule_ids`에 포함하라. 참고하지 않았다면 빈 배열."

**한계**: LLM이 인용을 누락할 가능성 존재 → Validation 경로가 이를 보완.

## 6. 측정 흐름 (1 케이스 처리)

```
1. ⑤ Consensus Engine
   ├─ 규칙 DB에서 RAG로 관련 규칙 top-K retrieve (K=5)
   │  (Solar Embedding 유사도 + category 매칭, hybrid)
   └─ Solar Pro 호출 준비: system_prompt + K개 규칙 텍스트 + tool 정의

2. Solar Pro 응답 수신
   └─ cited_rule_ids 추출 (function calling이므로 결정론적)

3. Validator 평가
   └─ K개 retrieve된 규칙 (또는 validator 보유 모든 규칙) 평가
      → 각 규칙에 Pass / Fail / N/A 라벨

4. rule_applied 이벤트 발행
   ├─ citation 집합 = cited_rule_ids (응답에서 추출)
   ├─ validation 집합 = {r | r.evaluation in {Pass, Fail}}
   └─ 각 rule_id에 대해:
       mode = "both" if (in citation ∩ validation)
              "citation" if (in citation only)
              "validation" if (in validation only)
       이벤트 1건 기록

5. 집계 갱신
   ├─ 규칙별 n_applications += 1 (unique case 수)
   └─ 케이스별 has_applied_rule = (적용 규칙 수 ≥ 1)
```

## 7. ErrorPrevention 이벤트 발생 조건 (구체화)

[크레딧_산정_휴리스틱.md](크레딧_산정_휴리스틱.md) §3.4 ErrorPrevention을 본 문서로 엄격화:

**ErrorPrevention 이벤트는 다음 셋 모두 충족 시 발생**:

1. `rule_applied{rule_id, case_id, mode: "both"}` — citation + validation 둘 다
2. validator 평가 결과가 **`Pass`** (Fail은 자동판정 자체 오류이므로 별도 흐름)
3. 해당 케이스에서 **관리자 override 미발생** (자동판정이 그대로 합의로 확정)

→ 원규칙 작성자(`R.author_expert_id`) + 원검수자(`R.source_diffs`의 reviewer들) 에게 ErrorPrevention 이벤트 1건씩 귀속.

→ 다중 귀속 시 가중치: 작성자 0.7, 각 source 검수자 0.3 / (검수자 수)로 분배.

## 8. KPI 측정 매핑

| KPI | 본 문서가 정의하는 것 | 산식 |
|---|---|---|
| **KPI 4-A** `rule_reuse_rate` | "1개 이상 규칙이 적용된 케이스" | `count(case where has_applied_rule) / total_cases` (윈도우별) |
| **Reusability** (휴리스틱 §3.3) | "한 규칙의 재적용 횟수" | `1 + min(4.0, log2(1 + n_applications))` |
| **ErrorPrevention** (휴리스틱 §3.4) | "후속 오류 방지 이벤트" | 위 §7 조건 충족 횟수 |

## 9. 본선 전 미결 사항

1. **Validator DSL v1.0 확장**: 산술식 평가, 다중 필드 cross-validation 지원
2. **RAG retrieval 설정**: K 값, 임계값, hybrid scoring 가중치 (1개월차 파일럿 후 보정)
3. **Citation 누락률 측정**: Solar Pro가 명시 규칙을 빠뜨리는 비율 → Validation 경로의 보완 효과 정량화
4. **Citation vs Validation 가중치 차등**: 현 휴리스틱은 동일 가중. 본선 전 분포 분석 후 차등 도입 검토
5. **ErrorPrevention 다중 귀속 가중치**: 0.7 / 0.3 분배는 휴리스틱 — 파일럿 데이터로 보정
