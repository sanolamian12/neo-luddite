# CrediGraph KPI 정의서

> 팀: 네오러다이트 · 프로젝트: CrediGraph
> 작성: 2026-05-17 · 본선 전 보정 1회 예정 (~2026-07)

본 문서는 본선 심사(8.5~8.13)에서 우리가 측정해 보고할 5개 KPI의 정의·산식·데이터 수집 포인트·목표값을 확정한다. 본선 슬라이드의 차트는 모두 이 문서에서 정의한 이벤트 스키마에서 자동 생성한다.

---

## 0. KPI 빅뷰

| # | KPI | 무엇을 묻는가 | 핵심 산식 | 목표값 |
|---|---|---|---|---|
| 1 | 문서 정형화 성공률 | 입력이 시스템에 잘 들어오는가 | 필수 필드 추출 성공 / 전체 입력 | ≥ 95% |
| 2 | 검수 운영 건전성 | 워크플로우가 굴러가는가 | 마감 내 완료 검수 / 배정 검수 + 불일치 탐지 정확도 | 100% + Precision ≥ 80%, Recall ≥ 70% |
| 3 | 합의 품질 | LLM+시스템이 옳은 답을 내는가 | 공개 유권해석 ground truth 일치율 외 | ≥ 80% |
| 4 | **시스템 학습 효과** | 우리 차별점이 작동하는가 | 후행 30건 vs 선행 30건의 품질 격차 | 유의한 우상향 추세 |
| 5 | **기여 정산 공정성** | 크레딧이 설명 가능·재현 가능한가 | 원장 추적률 + 이의제기 인용 처리율 | 추적률 100% / 인용 처리 ≥ 95% |

## KPI 1 — 문서 정형화 성공률 `parse_success_rate`

**정의**: Document Parse가 "에러 없이 돌아간 비율"이 아니라, **세무 도메인에서 다음 단계에 필요한 필수 키가 모두 추출된 비율**.

| 항목 | 내용 |
|---|---|
| 산식 | `필수 필드 N개가 모두 추출된 케이스 수 / 입력 100건` |
| 필수 필드 (본선 전 확정) | 신고 유형, 신고자(마스킹 ID), 신고기간, 세목, 과세표준, 산출세액, 첨부서류 목록 |
| 데이터 수집 포인트 | ② 문서 파이프라인 — `document_parse_result` 테이블에 필드별 추출 여부 기록 |
| 목표값 | ≥ 95% (제안서 약속) |
| 본선 가시화 | 100건 입력 → 95건 이상 모두 필수 필드 충족이라는 통과/실패 표 |

**근거**: Document Parse가 빈 결과를 반환해도 형식적으로는 에러 없음. 다음 단계(전문가 답안 작성)에 쓸 수 있는가가 진짜 기준.

## KPI 2 — 검수 운영 건전성 `review_pipeline_health`

파일럿 검수 사이클이 설계대로 굴러가는가. 두 개의 서브 지표.

### 2-A. 검수 완료율 `review_completion_rate`

| 항목 | 내용 |
|---|---|
| 산식 | `마감 내 완료 검수 수 / 배정 검수 총 수` |
| 분모 산정 | 100건 × (N-1)회 검수 = 케이스 1건당 검수 9회 (N=10) → 총 900회 |
| 데이터 수집 포인트 | ③ Expert Answer Pool + ④ Peer Review Graph (배정·완료 타임스탬프) |
| 목표값 | 100% (제안서 약속) |

### 2-B. 불일치 구간 탐지 정확도 `disagreement_detection_accuracy`

| 항목 | 내용 |
|---|---|
| Precision | 시스템이 탐지한 불일치 구간 중 관리자 검토에서도 불일치로 확정된 비율 |
| Recall | 관리자가 불일치로 본 구간 중 시스템도 탐지한 비율 |
| 데이터 수집 포인트 | ⑤ Consensus Engine 탐지 출력 + 관리자 검토 라벨 |
| 목표값 | Precision ≥ 80%, Recall ≥ 70% (본선 전 보정) |

## KPI 3 — 합의 품질 `consensus_quality`

Solar Pro 자동 판정 + 전문가 검수 + 관리자 검토가 결합되어 만든 합의안이 얼마나 옳은가. 세 개의 보조 지표.

### 3-A. Ground Truth 일치율 `gt_alignment_rate` ★ 핵심 지표

**공개 유권해석 문서를 ground truth로 활용** (제안서 §3.4 명시).

| 항목 | 내용 |
|---|---|
| 데이터 | 100건 중 ~30건을 공개 유권해석 사례로 구성 — 답이 이미 알려진 케이스 |
| 산식 | `시스템 합의안과 유권해석 답안의 핵심 결론 일치 건수 / GT 보유 케이스 수` |
| 핵심 결론 정의 | 과세 여부, 세액(±5% 허용), 적용 조항 — 셋 다 일치해야 "일치" |
| 목표값 | ≥ 80% |
| 본선 가시화 | "30건 GT 중 우리 시스템이 24건을 일치"라는 직관 메시지 |

### 3-B. 전문가 동의도 `expert_agreement_rate`

| 항목 | 내용 |
|---|---|
| 산식 | `합의안에 명시적 이의 없이 서명한 전문가 수 / 검수 참여 전문가 수` |
| 보조 통계 | Krippendorff's α (분류성 평가용 — 본선 후 옵션) |
| 목표값 | ≥ 90% (10명 중 9명 이상 동의) |

### 3-C. 관리자 수정률 `admin_override_rate`

| 항목 | 내용 |
|---|---|
| 산식 | `관리자 검토에서 수정된 자동 판정 건수 / 자동 판정 건수` |
| 해석 | 낮을수록 자동 판정 신뢰 ↑. 단 0%는 형식적 검토라는 신호 |
| 목표값 | 5% ≤ override ≤ 25% (건강한 범위) |

## KPI 4 — 시스템 학습 효과 `system_learning_effect` ★ 핵심 차별점

**정의**: 100건이 시간 순으로 처리되는 동안, 후행 케이스가 선행 케이스보다 더 효율적·정확하게 합의에 도달하는가. 모델 파라미터가 아닌 **시스템(프롬프트·RAG·규칙 DB·임계값)이 누적적으로 좋아지는가** 측정.

**윈도우 분할**: 100건을 시간 순으로 `early(1-30) / mid(31-70) / late(71-100)` 세 윈도우.

### 4-A. 재사용 규칙 적용률 `rule_reuse_rate`

| 항목 | 내용 |
|---|---|
| 정의 | ⑤ Consensus Engine이 기존 규칙 DB에서 1개 이상의 규칙을 참조한 비율 |
| 산식 | `규칙 참조가 발생한 케이스 수 / 윈도우 내 케이스 수` |
| 데이터 수집 포인트 | Consensus Engine RAG 호출 로그 + 규칙 DB 조회 ID |
| 목표값 | early: 0% (규칙 DB 비어있음) → late: ≥ 60% |
| 본선 가시화 | "1번째 케이스는 규칙 0개 → 100번째 케이스는 평균 4.2개 적용"이라는 누적 곡선 |

### 4-B. 검수 diff 감소율 `diff_volume_reduction`

| 항목 | 내용 |
|---|---|
| 정의 | 전문가들 답안 간 평균 diff 양이 줄어드는가 (수렴 속도) |
| 산식 | `mean(edit_distance(answer_i, answer_j))` 윈도우 평균 비교: `late_avg / early_avg` |
| 해석 | < 1 이면 후행 케이스가 더 빨리 수렴 = 시스템이 일관 가이드를 제공한다는 증거 |
| 목표값 | late/early ratio ≤ 0.75 (25% 이상 감소) |

### 4-C. 합의 도달 시간 단축 `time_to_consensus`

| 항목 | 내용 |
|---|---|
| 산식 | 케이스 시작 → 최종 합의 확정까지의 wall-clock 시간 윈도우 평균 |
| 목표값 | late/early ratio ≤ 0.7 |
| 주의 | 전문가 일정에 영향 받으므로 보조 지표 |

### 4-D. GT 일치율의 시계열 추세 `gt_alignment_trend`

| 항목 | 내용 |
|---|---|
| 정의 | KPI 3-A를 윈도우별로 분해 |
| 목표 | early 대비 late에서 GT 일치율 우상향 (회귀선 기울기 > 0, p < 0.05) |
| 본선 가시화 | 케이스 번호 x축, GT 일치율 y축 산점도 + 회귀선 |

> **본선 한 줄 메시지**: "시스템이 케이스를 처리할수록 규칙 적용률 0→60%↑, 전문가 간 diff 25%↓, GT 일치율 우상향. 모델은 그대로지만 시스템이 학습했다는 정량적 증거."

## KPI 5 — 기여 정산 공정성 `credit_fairness` ★ 핵심 차별점

크레딧 산정이 (a) 모든 기여 항목이 원장에 추적되고, (b) 산정 근거가 설명 가능하며, (c) 이의제기 절차가 실제로 작동하는가.

### 5-A. 원장 추적률 `ledger_traceability` ★ 반드시 100%

| 항목 | 내용 |
|---|---|
| 정의 | 발급된 모든 크레딧이 ⑥ Contribution Ledger entry로부터 역추적 가능한 비율 |
| 산식 | `원장 entry로 역추적 가능한 크레딧 건수 / 전체 크레딧 발급 건수` |
| 목표값 | **100%** (이 지표가 100% 미만이면 설계 결함) |
| 데이터 수집 포인트 | 크레딧 발급 시 `source_ledger_entry_hash` 필드 필수 기록 |
| 본선 가시화 | 임의 크레딧 한 줄 클릭 → SHA-256 해시 체인 따라 원본 기여 이벤트까지 추적 |

### 5-B. 산정 단계 분포 `scoring_stage_distribution`

| 항목 | 내용 |
|---|---|
| 정의 | 크레딧 산정이 어느 단계에서 확정되었는가 — 휴리스틱 / 백테스트 보정 / 관리자 최종 |
| 산식 | 단계별 카운트 비율 |
| 해석 | "휴리스틱 100%, 관리자 0%"는 검토 형식화 적신호. "관리자 100%"는 자동화 무력 |
| 목표값 | 휴리스틱 60~75%, 백테스트 보정 15~30%, 관리자 5~15% |

### 5-C. 이의제기 발생률·인용률 `dispute_rate` / `dispute_acceptance_rate`

| 항목 | 내용 |
|---|---|
| 발생률 산식 | `이의제기 건수 / 전체 크레딧 발급 건수` |
| 인용률 산식 | `실제 크레딧 조정으로 이어진 이의제기 / 전체 이의제기` |
| 목표값 | 발생률 ≤ 10%, **인용 처리율 ≥ 95%** (이의제기 시스템 작동성) |
| 무결성 검증 | 이의제기 → 조정 → 새 원장 entry → 새 크레딧 계산이 모두 append-only 기록 + 해시 체인 무결성 유지 |

### 5-D. 기여 유형별 크레딧 분포 `credit_type_distribution`

| 항목 | 내용 |
|---|---|
| 정의 | 초안 작성 / 검수 / 규칙 생성 / 오류 예방 4종 기여의 크레딧 비중 |
| 해석 | 한 유형이 90%를 가져가면 다른 기여 유형이 무의미 — 4종 분류의 정당성 검증 |
| 목표값 | 어느 한 유형도 60% 초과하지 않음 |

## 계측 인프라 — 이벤트 스키마

위 KPI들을 측정하려면 6개 컴포넌트에 다음 이벤트 로깅이 박혀 있어야 한다. **구현 시작 시점에 데이터 모델에 못박아둘 것.**

| 컴포넌트 | 필수 기록 이벤트 | 흐름 KPI |
|---|---|---|
| ② 문서 파이프라인 | `document_parsed{case_id, fields_extracted{}, parse_duration_ms}` | KPI 1 |
| ③ Expert Answer Pool | `assignment_created`, `answer_submitted{case_id, expert_id, timestamp, content_hash}` | KPI 2-A, 4-C |
| ④ Peer Review Graph | `review_completed{reviewer, target, diff_summary, edit_distance}` | KPI 2-A, 4-B |
| ⑤ Consensus Engine | `auto_judgment{case_id, rules_referenced[], confidence}`, `disagreement_detected{spans}`, `admin_override{before, after, reason}` | KPI 2-B, 3, 4-A, 4-D |
| Knowledge Asset Extractor | `rule_extracted{source_diffs[]}`, `rule_applied{rule_id, case_id}` | KPI 4-A |
| ⑥ Contribution Ledger | `contribution_recorded{type, expert_id, case_id, prev_hash}`, `credit_issued{source_ledger_entry_hash, amount, stage}`, `dispute_filed`, `dispute_resolved{adjustment}` | KPI 5 전체 |
| GT 검증 모듈 | `ground_truth_compared{case_id, system_answer, gt_answer, match}` | KPI 3-A, 4-D |

## 본선 슬라이드 매핑 (참고)

| 심사 항목 | 핵심 KPI |
|---|---|
| 타당성·논리성 | KPI 1, 2 (워크플로우 작동) |
| 기술성·탁월성 | KPI 3-A (GT 일치율) |
| 기술성·혁신성 | **KPI 4 전체** (시스템 학습 효과 — 우리만의 차별점) |
| 실용성·안전성 | **KPI 5-A** (원장 추적률 100%) + 5-C (이의제기 작동) |
| 시장성·실용성 | KPI 5-B, 5-D (정산 균형) |

## 본선 전 확정해야 할 미결 사항

1. KPI 1의 "필수 필드 7개" 최종 목록 (세무 전문가 자문 후 확정)
2. KPI 3-A의 공개 유권해석 ~30건 큐레이션 (도메인 다양성 확보)
3. KPI 2-B의 Precision/Recall 목표값 보정 (1개월차 파일럿 결과 기반)
4. KPI 4-A의 "규칙 1개 적용" 정의 — 직접 인용 / 의미 참조 / 후처리 검증 중 어느 단계까지 카운트할지
5. KPI 5-D 기여 유형 4종의 가중치 (휴리스틱 산정 공식)
