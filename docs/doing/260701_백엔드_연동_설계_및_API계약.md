프론트엔드(credigraph 프로토타입) ↔ 백엔드(세금 판례·규칙엔진)를 실사용 제품으로 연결하기 위한 계약서.
코드 착수 전에 이 문서로 인터페이스를 확정한다.

---

## 0. 전제 · 컴플라이언스

- **모델 호출은 오직 Seam A(`/api/chat`) 한 곳.** 나머지(B 플랫폼 CRUD, C 지식베이스)는 LLM 불필요.
- **Seam A는 Upstage만 사용** — LLM=Solar, 임베딩=Upstage embedding. 외산 LLM/임베딩/judge 사용 시 국내 트랙 위반.
- 백엔드 현황: LLM 호출 **0개**(순수 규칙기반). 즉 위반 소지는 오직 우리가 A를 구현할 때 생기며, Upstage로만 채우면 100% 충족.
- ⚠️ 프론트는 **Next.js 16(브레이킹 체인지 버전)**. 구현 시 `frontend/node_modules/next/dist/docs/` 먼저 확인(프론트 `AGENTS.md` 지시).

## 1. 아키텍처 — 3개의 이음새(seam)

```
실사용자 ──chat──▶  [A] AI 추론 백엔드  (Python/FastAPI + Upstage)   ★핵심
                      /api/chat → Solar + 판례 RAG + 규칙엔진(tool)
                            │ 대화 로그
세무감사자 ─audit──▶  [B] 플랫폼 영속 백엔드  (REST + DB)
                      ~28개 CRUD, 프론트 Zustand → fetch 교체
                            │ 참조자료
프론트 KB뷰어  ◀────  [C] 지식베이스 백엔드
                      backend/data 판례코퍼스 → KB 문서
```

- **A**: 채팅 답변을 실제 생성. 제품의 심장. Upstage가 여기 산다.
- **B**: 감사/정산/파이프라인 워크플로우의 영속화. LLM 없음. 데모 단계에선 목업 유지 가능.
- **C**: 백엔드가 수집한 판례를 감사자 참조자료(KB)로 노출.

배치: 백엔드는 별도 **FastAPI 서비스**(Python 엔진 재사용). 프론트는 `NEXT_PUBLIC_API_BASE` 또는 Next.js rewrite 프록시로 호출.

## 2. Seam A — `/api/chat` 계약 (가장 중요)

### 2.1 진실의 원천: 대화 스키마
프론트가 렌더링하는 assistant 메시지 구조 = 백엔드가 생성해서 내려줘야 할 정확한 모양.
출처: `frontend/lib/conversation-schema.ts`.

**Segment** (문장 단위, 주석 앵커라 `id` 고유·불변):
```
segment = { id, text, type, framework?, citations? }
  type ∈ context | question | ack | issue_framing | rule_statement |
         application | evidence_request | conclusion | caveat | follow_up
  framework ∈ 문언해석 | 목적론해석 | 체계적해석 | 실질과세원칙 |
              신의성실원칙 | 엄격해석 | 입증책임 | 유추해석      (8 canon)
  citations ∈ string[]   예: "소득세법 §35", "조심2025부4055"
```

**UI Block** (구조화 카드, discriminated union on `kind`):
```
verdict_card       = { kind, verdict, title, summary }
  verdict ∈ 전부인정 | 안분인정 | 부인 | 조건부
evidence_checklist = { kind, title, items:[{ label, required, note? }] }
```

**Message**: `{ id, role, order, segments[≥1], uiBlocks? }`

### 2.2 백엔드↔프론트 필드 매핑 (이미 1:1)
| 프론트 계약 | 백엔드 산출물 | 근거 |
|---|---|---|
| `verdict_card.verdict` | `clinic_expense_engine.ExpenseResult.verdict` | enum 4값 **완전 동일** |
| `verdict_card.summary/title` | `ExpenseResult.근거` | |
| `evidence_checklist.items[].label` | `ExpenseResult.필요증빙: list[str]` | |
| `segment.framework` | `analysis_reasoning.py` 8 canon | enum 동일 |
| `segment.citations` | 엔진 인용 + `backend/data` 판례 RAG | |
| `segment.type` | `analysis_structure.py` 결정문 구조 | |

→ **uiBlocks는 규칙엔진이 결정적으로 생성**, **segments(자연어 논증)는 Upstage가 RAG+엔진결과 기반으로 생성**. 이 하이브리드가 A의 핵심 설계.

### 2.3 직업 → 엔진 라우팅 (실측 반영)
프론트 실제 직업군: `frontend/lib/occupations.ts` → **`general | clinic | online-seller | beauty`**.
현재 **active는 `clinic` 하나뿐**, 나머지 3개는 "준비중(coming)".

| occupation | 상태 | 규칙엔진 | 처리 |
|---|---|---|---|
| `clinic` (병의원) | **active** | `clinic_expense_engine.evaluate(ClinicProfile, ExpenseInput)` | **MVP E2E 경로** ✅ |
| `online-seller` | coming | 없음 | 엔진 미구현 → RAG-only 또는 신규엔진 |
| `beauty` | coming | 없음 | 〃 |
| `general` | coming | 없음 | 〃 |

⚠️ **divergence 주의**: 백엔드 `creator_tax_engine`(크리에이터/유튜버)은 **프론트에 대응 직업이 없음**. 원 개발자 데모라인 산출물로, 지금은 미연결 자산. 향후 "creator" 직업 추가 시 재사용 가능(현재 계약 범위 밖).

→ 결론: **clinic 단일 경로로 A를 완성**한 뒤, coming 직업은 RAG-only로 확장, 필요 시 엔진 추가.

### 2.4 요청/응답 (확정)
**A-1 확정: 동기 단발**(non-streaming). 프론트가 `useExternalStoreRuntime`이라 후에 SSE 승격 가능하나 MVP는 단발.
**A-2 확정: LLM function-calling으로 엔진 입력 추출.** 대화에서 `ClinicProfile`/`ExpenseInput` 필드를 Solar가 채우고, 부족하면 되묻는다(별도 폼 없음).

**요청**
```jsonc
POST /api/chat
{
  "conversationId": "conv_clinic_golf_001",
  "occupation": "clinic",
  "history": [ /* 이전 Message[] (segments 텍스트/role/order) */ ],
  "userInput": { "text": "거래처와 골프를 쳤는데 접대비로 인정되나요?" }
}
```

**응답** (assistant `Message` 1개 = 2.1 스키마 그대로)
```jsonc
{
  "message": {
    "id": "asst_conv_clinic_golf_001_3",
    "role": "assistant",
    "order": 3,
    "segments": [
      { "id": "asst_..._3_s0", "text": "...", "type": "conclusion" },
      { "id": "asst_..._3_s1", "text": "...", "type": "rule_statement",
        "framework": "엄격해석", "citations": ["소득세법 §35"] },
      { "id": "asst_..._3_s2", "text": "...", "type": "evidence_request",
        "framework": "입증책임", "citations": ["조심2025부4055"] }
    ],
    "uiBlocks": [
      { "kind": "verdict_card", "verdict": "조건부", "title": "...", "summary": "..." },
      { "kind": "evidence_checklist", "title": "...", "items": [ { "label": "...", "required": true } ] }
    ]
  },
  "meta": {                          // 감사/디버깅용, 프론트 렌더에는 불필요
    "engine": "clinic_expense_engine",
    "extracted": { /* LLM이 채운 ClinicProfile+ExpenseInput */ },
    "ragCaseRefs": ["조심2025부4055"]
  }
}
```

**정보 부족 시** (필수 필드 미확보): `uiBlocks` 없이 `type:"follow_up"`/`"evidence_request"` segment만 담아 되물음. verdict는 정보 충족 후에만 생성.

### 2.5 내부 생성 파이프라인 (hybrid)
```
history+userInput
   │
   ▼  ① Solar(function-calling): 대화→ ClinicProfile/ExpenseInput 추출
   │      필수필드 부족? ──yes──▶ follow-up Message 반환(종료)
   │      no
   ▼  ② clinic_expense_engine.evaluate() → ExpenseResult(verdict,인정금액,리스크,근거,필요증빙)
   ▼  ③ Upstage 임베딩 RAG(backend/data 판례) → 관련 caseRefs
   ▼  ④ Solar: segments 작문(framework/citations 태깅, ②③ 근거 주입)
   ▼  ⑤ 매핑: ExpenseResult → verdict_card + evidence_checklist(uiBlocks)
   ▼  ⑥ assistant Message 조립·반환
```
- ②⑤는 **결정적**(엔진), ①④는 **LLM**. verdict는 항상 엔진이 권위 원천(할루시네이션 방지).

### 2.6 프론트 교체 지점
- `frontend/lib/runtime/use-replay-runtime.ts`(리플레이) → 신규 `use-remote-runtime.ts`(`/api/chat` 호출)로 교체.
- **segment id 규칙**(A-3 확정): `${message.id}_s${index}`, `message.id = asst_${conversationId}_${order}`. 결정적·전역고유(스키마 superRefine 통과).
- 세그먼트 렌더러·UI블록 컴포넌트 **무변경**.

### 2.7 RAG 코퍼스 순환 (하차장→검수→포장실→RAG→챗)
Seam A의 RAG 소스는 정적 판례코퍼스뿐 아니라 **Hub에서 검수 완료된 Q&A**로 계속 성장한다.
```
하차장(엑셀 Q+Upstage답변) → 작업실 검수 → 포장실 번들 → RAG 코퍼스(Upstage 임베딩)
                                                              │
                                    Seam C 지식베이스(/audit/knowledge) 와 공유
                                                              ▼
                                          Seam A /api/chat 그라운딩 (자기강화 루프)
```
- 포장실 번들 형태·청킹은 Upstage RAG 설계 확정 후(주최측 API 안내). 갭분석 §6.2 참조.

### 2.5 프론트 교체 지점 (구현 시)
- `frontend/lib/runtime/use-replay-runtime.ts`(리플레이) → `use-remote-runtime.ts`(실제 `/api/chat` 호출)로 교체.
- **컴포넌트/세그먼트 렌더러는 무변경** — 계약이 동일하므로.

## 3. Seam B — 플랫폼 REST 계약

프론트 `frontend/services/*.ts` 함수들은 이미 `async Promise<T>` 시그니처 + Zustand 목업.
`pool.ts:11` 주석: "동일 시그니처로 `fetch('/api/pool/...')` 교체" → **계약이 이미 코드에 있음**.

- 스키마 원천: `frontend/lib/poc-schema.ts`(Zod). 백엔드는 이 Zod와 동일 shape 반환.
- **정책 로직**(정산 배분, 크레딧 10점/승인 등)은 현재 프론트 서비스에 있음 → 백엔드 이전 시 권위 원천 이동 필요.
- 영속화 DB 미정(후보: Postgres). **데모까지는 목업 유지, 실사용은 B 구현** 전략 권장.
- 규칙: list 응답 = `{ items: Entity[], total }`. request는 클라 제공 필드만(서버생성 `id/createdAt/status기본/balanceAfter` 제외). 타입명은 poc-schema.ts 참조.

## 5. 권장 구현 순서 (설계 확정 후)

1. **A 계약 동결** — 2.4 결정사항(스트리밍/입력추출) 확정, 요청/응답 JSON 예시 고정.
2. **B 부록 완성** — 28개 엔드포인트 요청/응답 shape를 poc-schema에서 추출해 표로.
3. FastAPI 스캐폴드(`backend/api/`) + `clinic_expense_engine` 래핑 → `/api/chat` clinic 경로 1개 E2E.
4. Upstage Solar 연결(segments 생성) + Upstage 임베딩 RAG(판례 인용).
5. 프론트 런타임 교체 → clinic 대화 실연결 검증.
