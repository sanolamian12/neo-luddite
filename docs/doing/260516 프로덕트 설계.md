# 260516 프로덕트 설계

## 핵심 통찰: "우리가 최적화하는 것은 LLM이 아니라 시스템"

`API inference only, no fine-tuning`은 **약점이 아니라 의도된 설계**입니다. 우리 프로젝트는 **LLM 제품**이 아니라 **LLM을 활용한 합의 워크플로우 제품**이에요.

비유하자면 — ChatGPT 앱을 만드는 것과 비슷합니다. OpenAI의 GPT를 학습시키지 않지만, 앱은 점점 나아집니다. 프롬프트가 정교해지고, 검색 컨텍스트가 풍부해지고, 도구 호출이 영리해지면서요.

---

## 1. 우리가 실제로 빌딩하는 것 (LLM 빼고 보기)

LLM(Solar Pro)을 외부 의존성으로 빼고 보면, 우리 코드베이스는 다음 6개 컴포넌트입니다:

`┌──────────────────────────────────────────────────────────────┐
│  ① 비식별화 전처리기 (PII 마스킹 — 룰베이스 + 한국어 NER)    │
│     주민/사업자번호, 납세 내역, 이름·주소 마스킹             │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  ② 문서 파이프라인 (Upstage API 클라이언트 + 재시도/캐싱)    │
│     Document Parse → Information Extract → 정형 JSON         │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  ③ Expert Answer Pool (전문가 1차 답안 수집 + 라우팅 로직)   │
│     케이스 1건 × N명 독립 배정 / 마감 관리 / 답안 버저닝      │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  ④ Peer Review Graph (Neo4j 그래프 + diff 분석)              │
│     N-1 순환 검수 배정 / 수정 diff를 directed edge로 저장    │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  ⑤ Consensus Engine (Solar Pro 호출 + 3단계 워크플로우)      │
│     자동 판정(Solar Pro) → 관리자 검토 → 감사 로그           │
│     + 불일치 구간 탐지 / 고위험 케이스 분리                  │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  ⑥ Contribution Ledger + Credit Scoring (해시체인 + 대시보드)│
│     기여 유형별 append-only 로그 / 휴리스틱 → 백테스트 → 최종 │
└──────────────────────────────────────────────────────────────┘`

**여기서 Solar Pro API 호출은 ②와 ⑤에만 등장.** 나머지 ①③④⑥는 전부 우리가 짜는 **Python 코드, 데이터베이스 스키마, 알고리즘, UI** 입니다. 즉 LLM은 시스템의 한 부품이고, 코드베이스의 80%는 LLM과 무관한 백엔드 시스템입니다.

---

## 2. 그럼 무엇이 "점점 좋아지는가" — 시스템 레벨 러닝 루프

LLM 자체는 그대로지만, **시스템은 학습**합니다. 학습되는 자리는 5군데:

| 최적화 대상 | 어떻게 좋아지는가 | 산출물 |
| --- | --- | --- |
| **A. 프롬프트** | Solar Pro 호출 시의 system prompt·few-shot. 케이스별로 어떤 예시를 끼워넣을지 정교화 | Prompt 라이브러리 |
| **B. 입력 품질** | Document Parse + Information Extract 스키마 튜닝. 세무 도메인 키 추출 정확도 향상 | 도메인 스키마 |
| **C. 검색 컨텍스트 (RAG)** | 누적되는 합의 정답·기각된 답안을 벡터 인덱스에 쌓아 다음 케이스에 검색 컨텍스트로 주입 | 합의 케이스 벡터 DB |
| **D. 합의 알고리즘** | Peer Review Graph의 diff 패턴 분석 → 불일치 탐지 임계값·고위험 분리 규칙 튜닝 | 합의 알고리즘 파라미터 |
| **E. 재사용 규칙 (Knowledge Asset)** | 수정 이력에서 추출한 템플릿·체크리스트를 규칙 DB에 적재 → 다음 합의의 검색 컨텍스트나 후처리 검증에 활용 | 규칙 DB |

**핵심 사이클**:

`전문가 검수 → diff 수집 → Knowledge Asset 추출
                            ↓
           프롬프트·RAG·규칙 DB·임계값에 반영
                            ↓
            다음 케이스의 합의 품질 ↑
                            ↓
       (다시 전문가 검수가 봉합·교정) ↻`

이게 우리 제안서가 말하는 "**전문가 검수 자체를 학습 데이터로 전환**"의 실체입니다. 모델 가중치가 바뀌는 게 아니라, **모델을 둘러싼 컨텍스트와 워크플로우가 누적적으로 개선**됩니다.

## 3. KPI 재정의 — "LLM 정확도"가 아니라 "합의 시스템 산출물 품질"

본선에서 우리가 답해야 할 것은 "Solar Pro 정확도가 몇 %?"가 **아닙니다**. 우리 KPI는 다음과 같아야 합니다:

| 측정 대상 | KPI | 제안서 약속 |
| --- | --- | --- |
| **문서 정형화** | Document Parse 변환 성공률 | ≥ 95% (1개월차) |
| **검수 운영** | 파일럿 검수 완료율 / 불일치 구간 탐지 동작 | 100% (2개월차) |
| **합의 품질** | 전문가 동의도, 관리자 수정률 | 추가 정의 필요 |
| **시스템 학습 효과** | 후속 케이스에서의 재사용 규칙 적용률 | 제안서가 "측정한다"고만 함 — **본선 전 정의 필요** |
| **기여 정산 공정성** | 이의제기 발생률 / 관리자 조정률 | 추가 정의 필요 |

→ **본선 발표 슬라이드에서 "우리 LLM 점수"가 아니라 "우리 시스템이 만든 데이터셋·합의·정산의 품질" 을 보여주는 게 정답.**

---

## 4. 3개월 구현 = 위 6개 컴포넌트의 시간순 빌딩

CLAUDE.md의 월별 계획을 컴포넌트 시점으로 다시 읽으면:

| 기간 | 빌딩하는 컴포넌트 | "프로덕트로서" 무엇이 살아나는가 |
| --- | --- | --- |
| **1개월차** | ① 비식별화 + ② 문서 파이프라인 + ③ Expert Answer Pool 골격 | 세무 문서 100건이 정형 JSON으로 들어오고, 전문가에게 배정되는 데까지 동작 |
| **2개월차** | ④ Peer Review Graph + ⑤ Consensus Engine 프로토타입 | 전문가 10명이 답안 쓰고 서로 검수하면, 그래프가 쌓이고 Solar Pro가 합의 후보를 내는 데까지 동작 |
| **3개월차** | ⑥ Contribution Ledger + Credit Scoring + 대시보드 + API 마감 | 끝에서 끝까지 한 사이클이 돌고, 외부에서 REST API로 합의 엔진을 호출할 수 있고, 대시보드에서 기여·크레딧이 보임 |

산출물 3종(합의 데이터셋 / 대시보드 / 합의 엔진 API)은 이 6개 컴포넌트의 자연스러운 결과물입니다.

---

## 5. 그래서, 본선 데모에 무엇을 보여줘야 하는가

LLM 학습이 없으니까, 데모의 무게중심도 달라집니다:

- ❌ "우리 모델 정확도 X%" — 보일 필요도 없고 우리 차별점도 아님
- ✅ **"케이스 1건이 들어가서 합의 데이터 + 정산까지 나오는 전 과정 라이브 시연"**
- ✅ **"같은 케이스를 2번째로 돌렸을 때, 첫 번째 검수에서 추출된 규칙이 적용되어 결과가 다르게 나오는 모습"** ← 시스템 러닝 루프의 가시화
- ✅ **"기여 원장의 해시체인이 위변조 시 깨지는 데모"** ← 안전성 심사 항목 직격
- ✅ **"전문가 A·B·C의 diff가 그래프로 시각화되는 화면"** ← 혁신성 심사 항목 직격

---

## 한 줄 정리

**우리가 만드는 것은 "더 똑똑한 LLM"이 아니라, "전문가와 LLM이 함께 일하면서 그 과정 자체가 데이터·자산·크레딧으로 누적되는 시스템"입니다. LLM은 그 안의 호출 가능한 부품일 뿐이고, 코드베이스의 80%는 그 부품을 둘러싼 워크플로우·그래프·원장·대시보드입니다.**
