## 3. 데이터 저장

| 종류 | 결정 | 담는 것 |
|---|---|---|
| **관계형 DB** | **PostgreSQL 15** | 케이스, 답안, 검수, 사용자, RBAC, 크레딧 원장 |

| **벡터 DB** | **pgvector** (Postgres extension) ★ 신규 결정 | 합의 케이스 임베딩, 규칙 DB 검색 |
| **객체 저장소** | **S3 호환 (대회 제공)** | 원본 PDF/스캔, 비식별화 전/후 |

### pgvector 선택 이유

- 별도 Qdrant/Weaviate 운영 인스턴스 추가 안 함 → 본선 일정에 인프라 면적 최소화
- 100건 케이스 + 규칙 수십 건 규모는 pgvector로 충분 (HNSW 인덱스로 ms 단위 검색)
- Postgres 백업·운영과 동일 흐름으로 묶여 무결성 검증·복구가 단순
- 본선 후 1,000건 이상 확장 시 Qdrant로 교체 가능 (스키마 유지)

### "벡터 DB가 왜 필요한가" — 한 줄 설명

일반 DB는 *정확히 일치하는 행*을 찾지만, 벡터 DB는 *의미가 가까운 행*을 찾는다. Consensus Engine이 새 케이스를 처리할 때 누적 합의 케이스·규칙 중 "이 케이스와 의미적으로 비슷한 top-5"를 꺼내 Solar Pro에 컨텍스트로 주입하는 RAG의 핵심 장치 ([260516 프로덕트 설계 §2-C](../260516%20프로덕트%20설계.md)).

## 6. PII 비식별화 (컴포넌트 ①) — 2-레이어 안전 설계

[HF 사용정책 §4-4](HF_사용정책_국내AI트랙.md) 회색지대(국산 공개 NER 모델) 회피.

### 결정

| 레이어 | 도구 | 처리 항목 |
|---|---|---|
| **1차 룰베이스** | 정규식 (결정론적, 트랙 무관) | 주민/사업자/계좌/카드번호 — 형식 정해진 PII |
| **2차 의미 기반** | **Upstage Information Extract** (스키마 기반 키 추출) | 이름 / 주소 / 전화번호 — 문맥 필요 PII |
| **3차 (옵션, 보류)** | 한국어 NER 외부 모델 | 워크숍 답변 확인 후 도입 검토 — 본선 스코프 외 |

→ 1차+2차로 본선 100건 커버. 트랙 위반 위험 없음.

### 한국어 NER이 무엇이며 왜 우리는 미도입했는가

NER(Named Entity Recognition) = 문장에서 "이건 사람 이름, 저건 주소, 저건 회사명"을 분류하는 AI 기술. 이름·주소처럼 정규식으로 못 잡는 PII에 필요. KLUE-BERT / KoBERT 등 국내 한국어 모델이 일반적이지만 5개 연계기업산이 아니라 트랙 회색지대 → **Information Extract로 NER 기능을 대체**.

## 3. 환경 토폴로지

### 3.1 개발 환경 (각자 로컬)

```
[개발자 PC]
 └─ docker-compose up
     ├─ api (FastAPI)
     ├─ worker (Celery)
     ├─ postgres + pgvector
     ├─ neo4j
     ├─ redis
     └─ web (Next.js dev)
외부 의존: Upstage API, S3 (개발용 버킷)
```

### 3.2 운영/데모 환경 (대회 제공 A100 인스턴스 단일 노드)

```
[대회 GPU 인스턴스 — A100 × 1]                  [Upstage API]
 ├─ Postgres + pgvector  ←──────────┐
 ├─ Neo4j (Community)               │
 ├─ Redis                           │
 ├─ FastAPI (Gunicorn 2 worker)     │  HTTPS
 ├─ Celery worker × 2               ├─────────→ Solar Pro 3
 ├─ Next.js (PM2 or container)      │           Document Parse
 ├─ Nginx (TLS 종단, 라우팅)        │           Information Extract
 ├─ GPU 활용 잡 (§4)                │           Solar Embedding
 └─ S3 마운트 (대회 제공 스토리지)  │
                                    │
[관리자/전문가/사용자 브라우저]─────┘

URL: 1개 도메인 + 서브패스
  / (채팅) /expert (검수) /admin (대시보드) /api (REST)
```

**원칙**: 본선 3개월 일정에 멀티노드 운영은 면적만 키운다. **단일 노드 docker-compose 콜로케이션** 으로 단순화.

### 3.3 백업

- Postgres: daily `pg_dump` → S3, 7일 보존
- Neo4j: daily online backup → S3
- 원본 PDF: S3 버전ing 활성화
- 별도 리전 백업 버킷 (§2 우선순위 ★★★)

## 4. A100 GPU 활용 시나리오 — 의도적 명분

API 추론에 GPU가 안 쓰이므로 다음 워크로드로 GPU를 활용한다 (트랙 준수).

### 4.1 (메인) Document Parse 후처리 + 임베딩 캐시 빌드 배치
Solar Embedding API RPS 한도 처리를 위해 배치 큐를 worker에서 병렬화. 머신 활용 (직접 GPU 추론 아님).

### 4.2 (옵션) Peer Review Graph diff 분석 — 의미 diff 통계
검수 diff 텍스트들의 임베딩 유사도 분포 분석 (KPI 4-B `diff_volume_reduction` 산출). PyTorch + Solar Embedding 결과를 GPU 메모리에 올려 코사인 매트릭스 계산. **데이터 도구로서의 PyTorch 사용** — 모델 추론 아님 → 트랙 OK ([HF_사용정책 §0](HF_사용정책_국내AI트랙.md)).

### 4.3 (옵션, 워크숍 후) 한국어 NER 로컬 추론
5/21 워크숍에서 국산 공개 모델 인정 시 → KLUE-NER 류를 A100에 로드해 PII 마스킹 2차 보완. **본선 스코프 외**.

→ A100을 "정량 분석 / 배치 / 옵션 NER" 머신으로 포지셔닝. 심사 질의 시 §4.1+4.2로 응답.

| 원본 세무 문서 PDF × 100건 | ~2GB |
| Document Parse HTML/Markdown 결과 | ~20MB |
| Information Extract JSON | ~5MB |
| Postgres (답안/검수/원장/크레딧 + 감사 로그) | ~5GB |
| Neo4j (전문가 10 × 케이스 100 × 검수 9 edge + diff payload) | ~3GB |
| pgvector 인덱스 | <1GB |
| Solar Embedding API 응답 캐시 | ~1GB |
| 시스템 로그 / 백업 | ~5GB |
| Docker 이미지 + OS + 빌드 산출물 | ~30GB |
| **합계** | **~50GB** (300GB의 17%, 80% 여유) |

## 6. Upstage API 호출량 예상

| 단계 | 호출 수 (100건 기준) | 비고 |
|---|---|---|
| Document Parse | 100 | 케이스당 1회 |
| Information Extract | 100~200 | 필수필드 7개 추출 + PII 2차 |
| Solar Embedding (인덱싱) | 100 + 규칙 ~25 | RAG 인덱스 |
| Solar Pro (합의 자동판정) | 100~300 | 케이스당 1~3회 (불일치 재호출 포함) |
| Solar Pro (채팅 UI 데모) | 일일 수십 회 | 본선·결선 데모 시연 |
| **전체** | **~700~1,000 호출** | 응답 캐시(24h)로 회귀 테스트 재호출 방지 |

## 7. 보안 · 컴플라이언스 인프라

- 비식별화 모듈은 **파이프라인 첫 노드 + 출구 검증** 둘 다
- 모든 API 응답 로그에 PII 필드는 마스킹된 형태로만 기록 (structlog 필터)
- Contribution Ledger 테이블에 row-level append-only DB 트리거 (UPDATE/DELETE 차단)
- 매일 해시 체인 무결성 검증 잡 (Celery beat)
