## 1. 한 줄 요약

대회가 **GPU 서버 1대 + 스토리지 300GB + Upstage API 호출**을 **1,000 크레딧(약 1천만원) 패키지**. 모든 시스템을 그 서버 1대에 함께 올리는 단순 구성. 잔여 예산 **약 300만원**으로 백업·도메인·결선 예비비 등을 확보.

| 자원 | 제공량 | 우리 활용 |
| --- | --- | --- |
| **GPU A100 1대 (클라우드 인스턴스)** | 2개월 | 메인 서버. 데이터베이스·웹·API 모두 이 위에 올림 |
| **스토리지** | 300GB (S3 호환) | 원본 PDF·백업·로그 등. 실제 사용량 ~50GB 예상 |

→ GPU 시간·스토리지·네트워크·API 호출이 모두 1,000 크레딧 풀에서 빠지는 구조. 팀에서 알아서 분배.

### A100 GPU 활용에 관한 솔직한 짚어두기

Upstage 가 외부 API라 사실 우리는 **GPU를 직접 추론에 거의 안 씀**. 그래도 다음 3가지로 의도적으로 활용해 명분을 둠:

1. 문서 변환·임베딩 작업의 병렬 처리 (서버 머신으로서)
2. 검수 diff 텍스트 유사도 분석 (KPI 측정용 정량 분석)

## 4. 어디에 어떻게 올라가나 ─ 단순 구성도

```
┌──────────────────────────────────────────────────┐
│  대회 제공 클라우드 인스턴스 1대 (A100 GPU)      │
│                                                  │
│  ├─ 백엔드 서버 (Python · FastAPI)               │
│  ├─ 백그라운드 작업자 (Celery)                   │
│  ├─ 데이터베이스 4종 (Postgres·Neo4j·Redis…)     │
│  ├─ 웹사이트 (Next.js)                           │
│  └─ 외부 노출 (Nginx + HTTPS)                    │
│             │           │                        │
│             │           └──→ Upstage API (외부)  │
│             └──→ S3 스토리지 (대회 제공)         │
└──────────────────────────────────────────────────┘
                    ↓ HTTPS
        본선 데모용 URL 1개 (예: credigraph.kr)
        ├─ /        (일반 사용자 채팅)
        ├─ /expert  (전문가 검수)
        ├─ /admin   (관리자 대시보드)
        └─ /api     (외부 호출자용 REST API)
```

**원칙**: 본선 3개월 일정에 서버를 여러 대로 쪼개면 운영 부담만 커짐. **1대에 모두 올리고 단순화**. 본선 후 트래픽이 늘면 분리 검토.

| 부분 | 대응되는 컴포넌트 | 의미 |
| --- | --- | --- |
| **Credi** | ⑥ Contribution Ledger + **Credit** Scoring | 전문가 기여를 **크레딧으로 정산**하는 시스템. 일회성 외주 검수가 아니라 기여가 지속 인정되는 구조 (제안서 §1.7) |
| **Graph** | ④ Peer Review **Graph** | 전문가들의 상호 검수 관계를 방향 그래프로 저장. 케이스 1건 × N명 × N-1회 검수 = N² 감독 신호 (제안서 §2.6) |

부수적으로 **Credi-** 에는 *Credibility(공신력)* 의 뉘앙스도 들어 있어 보입니다. SHA-256 해시 체인으로 위변조 방지 + 합의 기반 신뢰 확보라는 또 다른 축과도 맞습니다 (제안서 §2.6의 "감사 추적 가능성").

즉 한 단어 안에 **"전문가 기여를 크레딧으로 정산하는 그래프 기반 합의 시스템"** 이라는 프로젝트 정체성이 압축되어 있는 작명입니다.

## 5. 데이터가 얼마나 쌓이나 ─ 300GB 중 ~50GB 사용

| 항목 | 추정 용량 |
| --- | --- |
| 원본 세무 문서 PDF (100건) | ~2GB |
| AI 변환 결과·정형 JSON | ~25MB |
| PostgreSQL (케이스·답안·원장 등) | ~5GB |
| Neo4j (검수 관계 그래프) | ~3GB |
| 의미 검색 인덱스 (pgvector) | <1GB |
| AI 응답 캐시 | ~1GB |
| 로그·백업 | ~5GB |
| OS·도커 이미지 | ~30GB |
| **합계** | **~50GB** (전체의 17%) |

→ 본선 후 1,000건으로 확장해도 충분한 여유.

## 6. Upstage API 호출이 얼마나 발생하나 (100건 기준)

| 단계 | 호출 수 | 비고 |
| --- | --- | --- |
| Document Parse 
(PDF 정형 변환) | 100 | 케이스당 1회 |
| Information Extract 
(필수 필드 7 + 개인정보 추출) | 100~200 |  |
| Solar Embedding 
(검색 인덱싱) | ~125 | 케이스 100 + 규칙 ~25 |
| Solar Pro 
(AI 합의 자동 판정) | 100~300 | 케이스당 1~3회 (불일치 시 재호출 포함) |
| Solar Pro 
(채팅 UI 데모) | 일일 수십 회 | 본선·결선 시연용 |
| **전체** | **약 700~1,000 호출** | 같은 요청은 24시간 캐시로 재호출 방지 |

→ 1,000 크레딧 패키지로 본선 100건 + 데모 충분히 커버. 결선용 추가 크레딧은 §3에서 100만원 배정.

## 7. 보안·백업 운영

| 항목 | 운영 방식 |
| --- | --- |
| 데이터베이스 백업 | 매일 자동 백업 → S3 (7일 보존) |
| 원본 PDF | S3 버전 관리 활성화 (실수로 덮어써도 복구 가능) |
| 별도 리전 백업 | 본선 직전 인스턴스 장애 대비 (§3 ★★★) |
| 감사 로그 | 모든 API 응답에 개인정보는 마스킹된 형태로만 기록 |
| 원장 위변조 방지 | 데이터베이스 차원에서 수정·삭제 차단 + 매일 무결성 검증 |
