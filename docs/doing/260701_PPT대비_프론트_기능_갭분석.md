# PPT(CrediGraph Hub 기획) 대비 프론트엔드 기능 갭 분석

작성일: 2026-07-01 · 대상: `docs/네오 러다이트 - 전체 서비스 구상안_260624_정중은.pptx`(49슬라이드) ↔ `frontend/`(credigraph 프로토타입)

## 0. 핵심 결론
- PPT는 **CrediGraph "Hub"** = 창고 메타포의 관리자/작업자 **데이터생산·검수·정산 플랫폼**(30+화면) 상세 기획서.
- 프론트의 `/admin` + `/audit` 영역이 이 Hub에 해당. **워크플로우 구조는 ~80% 존재**, 단 **용어(메타포)가 다름**(하차장/화물/일감/검수/포장/상품/박스 ↔ pool/task/audit/review/pipeline/batch/version).
- 추가로 프론트에는 PPT에 거의 없는 **엔드유저 AI 서비스(`/chat`)**와 **지식베이스(`/audit/knowledge`)**가 이미 있음(= CrediGraph Service / LAB 방향).
- **보완 필요 핵심 3가지**: ① 하차장 화물 업로드(intake), ② 포장실 상품/박스/발송 모델, ③ 마이페이지·기여통계·자격/통장 파일 등 부수 화면.

## 1. 용어 매핑 (PPT 메타포 ↔ 프론트 엔티티)
| PPT(창고) | 프론트 | 비고 |
|---|---|---|
| 화물 Cargo (질문+AI답변 세트) | conversation / PoolCandidate | **생성 경로 다름**(업로드 vs 챗 자동등록) |
| 일감 Task | AuditTask | 조건필터 구조 거의 동일 |
| 결과물 Result | Audit(+LineFeedback) | |
| 기여 Contribution | 승인된 feedback → LedgerEntry | |
| 상품/박스 Product/Box | TrainingBatch/ModelVersion | **모델 다름**(데이터상품 vs ML버전) |
| 발송 Shipping(→AI사전DB) | version promote/deploy | 개념 유사, 화면 없음 |

| 우편 Mail / 문의 Inquiry | Mail / Inquiry | |

## 2. Hub 방(room)별 커버리지

### ✅ 상황실 (관리자 대시보드/문의/우편) — 존재
- PPT: 대시보드(방별 KPI), 문의목록/상세, 우편발송함/상세/발송하기
- 프론트: `/admin/dashboard`, `/admin/inquiries`, `/admin/mail`
- 갭(경미): **문의 상세** 전용 라우트 없음(`inquiries/[id]` 부재 → 인라인 처리 추정), **우편 발송하기(MDEditor 작성)** 전용 화면 없음(`mail` 단일 페이지).

### ⚠️ 하차장 (화물 intake) — 구조적 갭
- PPT: 화물 목록/**등록(질문·AI답변 파일 .txt/.md/pdf 업로드 + 분류)**/상세
- 프론트: `/admin/pool`, `/admin/pool/[conversationId]` (챗에서 자동 등록된 대화 풀)
- **갭(중요)**: 관리자가 Q+AI답변 세트를 **업로드해 화물 생성하는 화면 부재**. 프론트는 "실사용자 챗 → 자동 pool" 모델뿐. LAB/외부 산출물을 배치 투입하려면 **업로드 intake 보완 필요**. (분류=일반/소상공인/병의원은 occupation과 일치 ✅)

### ✅ 작업실 (일감) — 강한 커버리지
- PPT 관리자: 일감 목록/상세/등록(모집인원·조건필터·마감일)
- PPT 작업자: 일감목록/내일감목록/일감상세(작업전·후)/**작업공간(문장별 코멘트)**
- 프론트 관리자: `/admin/tasks`(+`/new`,`/[taskId]`) / 작업자: `/audit/queue`(+`/[taskId]`), `/audit/[conversationId]`(작업공간)
- 일치도 높음: AuditTask 조건(minAcceptedContributions·minCategoryExperience·rankingPercentile)이 PPT의 "참여횟수/기여도/랭킹 필터"와 **정확히 대응** ✅
- 갭(경미): "내 일감 목록" 별도 뷰 명확치 않음(`/audit/work`가 부분 대체).

### ✅ 검수실 (결과물 검수) — 강한 커버리지
- PPT: 결과물 목록/상세(검수전·후)/**검수모드(코멘트별 기여 인정 체크, 디폴트 all)**, 작업자 문의하기
- 프론트: `/admin/inspection`(+`/[auditId]`), `/audit/work`,`/audit/results`(+`/[auditId]`)
- `review.setDecision(feedbackId, accepted)` = PPT 검수모드와 **정확히 대응** ✅

### ⚠️ 포장실 (상품/박스/발송) — 모델 불일치 (최대 갭)
- PPT: **상품(Product)** 목록/상세/**포장하기(상품→박스 묶음)**, **박스(Box)** 발송목록/상세(→AI 사전DB 덤프). 작업자 read-only 상품·발송 조회.
- 프론트: `/admin/pipeline`(batches/versions) = 승인피드백→**TrainingBatch→PR/CI(mock)→ModelVersion→promote/rollback**
- **갭(중요)**: 둘 다 "승인 기여물을 상품화·버전화"지만 **객체모델이 다름**. 프론트에 **상품/박스/발송 엔티티·화면 없음**, 작업자용 상품·발송 read-only 뷰 없음. → **개념 정합화 결정 필요**(PPT의 데이터상품 패키징 모델 채택 vs 프론트 ML파이프라인 유지).

### ✅ 기여 통장 (작업자) — 존재
- PPT: 대시보드/기여목록/기여상세(인정항목만 표시)/우편수신함
- 프론트: `/audit/dashboard`,`/audit/ledger`,`/audit/results/[auditId]`,`/audit/mailbox` ✅

### ✅ 작업자 관리 — 존재(부수 갭)
- PPT: 작업자 목록/상세/등록, **자격증명 PDF·통장사본 파일**, **기여 이력/정산 통계 그래프**
- 프론트: `/admin/auditors`(+`/new`,`/[auditorId]`) ✅
- 갭: **자격증명·통장 파일 업로드 부재**(AuditorEntry.qualifications는 문자열배열뿐), **기여통계 전용 그래프 화면 부재**(auditor.stats 데이터는 있음, dashboard 부분 포함 가능).

### ✅ 정산 관리 — 존재(경미 갭)
- PPT: 정산목록/새정산대장(AI매출×비율=배당총액 자동)/정산상세(작업자별 지급액, 출력)
- 프론트: `/admin/settlement`(+`/new`) ✅ (even/weighted_by_count 일치)
- 갭(경미): "AI매출×비율" 유도항목 미모델(프론트는 pool 금액 직접), 지급현황(N/M명) 추적·출력(export) 경미.

### ⚠️ 공통 — 마이페이지 갭
- PPT: 메인/로그인(관리자·작업자 선택), **내 정보(계정정보/비번변경/탈퇴)**
- 프론트: `/`,`/login`,`/select`(역할선택 로그인 존재)
- **갭**: **내 정보(마이페이지) 계정설정 화면 부재**(비밀번호 변경·탈퇴).

## 3. 프론트에 있으나 PPT에 없는 것 (추가 자산)
- `/chat/[occupation]` — 엔드유저 AI 세무상담(= CrediGraph **Service**). PPT는 Hub만 상세, Service는 미상세.
- `/audit/knowledge`(+new/edit) — **지식베이스**. PPT에 없음. → Seam C(판례 KB)와 연결되는 추가 자산.
- `/audit/chat-logs` — 대화 리플레이 로그.

## 5. 제품 방향 결정 (2026-07-01 확정)
- [x] **하차장 intake = 엑셀 업로드 방식.** 관리자가 .xlsx 업로드(A열=질문, B열=Upstage 답변, ~100행). **각 행 = 1개 작업(화물)**. 관리자가 행별로 **일감화(데이터 종류·참여 기간·참여 인원 설정)**하여 작업실로 넘김. → §6.1 상세.
- [x] **포장실 = RAG 번들.** 하차장 100건이 검수 완료되면 그 단위로 **묶어 RAG를 형성하는 쪽으로 export**. 프론트의 ML batch/version 모델이 아니라 **RAG 코퍼스 생성** 방향. 정확한 형태는 **Upstage AI+RAG 구성 방식에 의존**(주최측 API 안내 수령 후 확정). → §6.2.
- [x] **내 정보 화면 = 생략.** 별도 마이페이지 대신 **대시보드에 "어서오세요 OOO 님 (아이디)" + [관리자]/[세무사] 라벨**만 표기. (비번변경·탈퇴 화면 불필요)

## 6. 보완 상세 스펙

### 6.1 하차장 — 엑셀 화물 업로드 & 행별 일감화 (관리자)
**흐름**
```
① .xlsx 업로드 (A열 질문 · B열 Upstage 답변 · ~100행)
   ▼ 파싱
② 미리보기 테이블 (행 = 질문 | 답변 | 분류 지정)
   - 각 행 → Conversation 1건 생성:
       user 메시지(질문) + assistant 메시지(답변)
       assistant 답변은 문장단위 절단 → segments (참고: 슬라이드1 "유진이가 이미 함")
   - 각 행 → PoolCandidate 1건 (occupation=분류)
   ▼ 분류 지정 (행별 또는 일괄): 일반/병의원/온라인셀러/미용
③ 일감화: 행 선택 → AuditTask 생성
     - 참여 기간(deadline) · 참여 인원(capacity) · 조건(conditions) 설정
     - 선택 conversationIds 로 Task 등록 → 작업실(/audit/queue)로 이동
```
**화면**: `/admin/pool` 에 **"엑셀 업로드"** 진입 + 신규 `/admin/pool/upload`(파싱·미리보기·분류·일감화). 기존 `/admin/tasks/new`(일감 등록)와 연계 또는 통합.
**데이터 영향**: 신규 엔티티 불필요 — 기존 Conversation/PoolCandidate/AuditTask 재사용. 신규 로직 = **xlsx 파싱 + 답변 문장분할(segment 생성)**.
**주의**: 엑셀 답변은 plain text라 초기 segment는 `type` 기본값·framework/citation/uiBlocks 없음. framework·citation·검증은 **작업자 검수 단계**에서 부여되거나 Seam A 생성분에만 존재.

### 6.2 포장실 — RAG 번들 export (관리자)
**방향**: 검수 완료된 결과물(승인된 문장별 코멘트 포함)을 **배치 단위로 묶어 RAG 코퍼스 생성** → Upstage 임베딩으로 인덱싱 → Seam A `/api/chat`의 RAG 소스로 순환.
```
검수완료 Audit N건 → 번들(Box) → RAG 코퍼스(청크: 검증된 Q&A + 문장 코멘트/근거)
                                   → Upstage embedding 인덱스 → /api/chat 그라운딩
```
**의존**: 청킹 단위·인덱스 포맷·임베딩 엔드포인트는 **Upstage RAG 설계 확정 후**(주최측 API 안내). 그 전까지 화면은 "번들 생성 + export" 골격만.
**프론트 조정**: 기존 `/admin/pipeline`(batch/version)을 **RAG 번들 관점으로 재해석**하거나 축소. ML promote/rollback 개념은 실사용 초기엔 불요.
**연결**: 이 RAG 코퍼스 = Seam C 지식베이스(`/audit/knowledge`) + Seam A RAG의 **공유 코퍼스**.
