# 크레딧 산정 휴리스틱 공식 v0.1

> 팀: 네오러다이트 · 프로젝트: CrediGraph
> 작성: 2026-05-17 · 본선 전 파일럿 보정 1회 예정

본 문서는 [KPI_정의.md](KPI_정의.md) §끝 미결사항 #5의 해소안. 기여 4종(초안/검수/규칙/오류예방) × 가중치 4차원(품질/난도/재사용성/오류예방효과)의 휴리스틱 산정 공식을 확정한다. 이는 v0.1로, 100건 파일럿 결과 기반 보정 단계(Stage 2~3)에서 갱신된다.

---

## 1. 산정 공식 (요약)

```
Credit(contribution) = Base[type] × Quality × Difficulty × Reusability × ErrorPrevention
```

| 변수 | 적용 유형 | 범위 |
|---|---|---|
| Base | 4종 모두 (유형별 다른 값) | 고정 정수 |
| Quality | 4종 모두 | [0.5, 1.5] |
| Difficulty | 4종 모두 | [0.8, 2.0] |
| Reusability | Rule만 (나머지는 1.0 고정) | [1.0, 5.0] |
| ErrorPrevention | Draft·Review·Rule (D 유형 본인은 1.0 고정) | [1.0, 3.0] |

---

## 2. 4종 기여 유형 정의 및 Base 값

| 유형 | 작동 정의 | 측정 단위 | Base |
|---|---|---|---|
| **A. Draft (초안)** | Expert Answer Pool에 1차 답안 제출 | 케이스 × 전문가 = 1회 | **60** |
| **B. Review (검수)** | Peer Review Graph에 diff·코멘트 제출 (순환 검수 1회) | 케이스 × 전문가 × 대상답안 = 1회 | **40** |
| **C. Rule (규칙 생성)** | Knowledge Asset Extractor가 추출한 후보를 전문가가 승인·정제한 1건 (또는 직접 등록) | 규칙 1건 | **200** |
| **D. ErrorPrevention (오류 예방)** | 후속 케이스의 자동판정이 과거 기여(검수/규칙)를 참조 + 관리자 override 미발생 → 원기여자에게 1회 귀속 | 이벤트 1건 | **100** |

**Base 값 비율 근거**:
- Draft를 60으로 잡고 나머지를 상대 비율로 설정
- Review는 Draft의 2/3 (빈 캔버스 없음, 그러나 정확성 요구)
- Rule은 1회 등록으로 다회 재사용되므로 Base 3.3배 + Reusability 곱셈
- ErrorPrevention은 간접 귀속이라 명확한 단일 이벤트로 평가

## 3. 가중치 차원별 산식

### 3.1 Quality `∈ [0.5, 1.5]` — 이벤트 기반

| 트리거 | 값 | 기록 시점 |
|---|---|---|
| 합의안 그대로 채택 | × 1.2 | Stage 2 (합의 확정 시) |
| 부분 채택 / 큰 수정 발생 | × 0.7 | Stage 2 |
| 관리자 우수 표시 | × 1.5 | Stage 3 |
| 기각 / 무효 판정 | × 0.5 | Stage 2 |
| 기본 (이벤트 없음) | × 1.0 | Stage 1 default |

### 3.2 Difficulty `∈ [0.8, 2.0]` — 자동 계산

케이스 메타데이터에서 자동 산정. 모든 유형의 기여에 동일 적용.

```python
d_raw = 0.3 * num_articles            # 적용 조항 수
      + 0.2 * num_attachments          # 첨부서류 수
      + 0.3 * log10(1 + tax_base_krw)  # 과세 표준 규모 (로그)
      + 0.2 * exception_flag           # 예외 케이스 여부 (0/1)

Difficulty = clip(0.8 + 1.2 * percentile(d_raw, all_100_cases), 0.8, 2.0)
```

| 변수 | 출처 | 비고 |
|---|---|---|
| num_articles | Information Extract 결과 | 케이스에 인용된 세법 조항 수 |
| num_attachments | Document Parse 결과 | 첨부서류 개수 |
| tax_base_krw | Information Extract 결과 | 과세 표준 (KRW) |
| exception_flag | 관리자/전문가 태깅 | 예외 케이스 = 1 |

**가중치 0.3/0.2/0.3/0.2는 휴리스틱** — 100건 파일럿 후 회귀 분석으로 보정.

### 3.3 Reusability `∈ [1.0, 5.0]` — Rule 유형만

규칙이 후속 케이스의 ⑤ Consensus Engine에서 재참조될 때마다 갱신.

```python
Reusability = 1 + min(4.0, log2(1 + n_applications))

# n=0:  1.0  (등록 시점)
# n=1:  2.0
# n=3:  3.0
# n=7:  4.0
# n=15: 5.0  (cap)
```

다른 유형(Draft/Review/ErrorPrevention)은 Reusability = 1.0 고정.

### 3.4 ErrorPrevention `∈ [1.0, 3.0]` — 이벤트 가산

후속 케이스에서 본 기여가 자동판정에 직접 참조 + 관리자 override 미발생 시 +1.0 이벤트.

```python
ErrorPrevention = 1.0 + min(2.0, 0.5 * n_prevention_events)

# n=0: 1.0
# n=2: 2.0
# n=4: 3.0 (cap)
```

| 적용 | 비적용 |
|---|---|
| Draft, Review, Rule | ErrorPrevention 유형 본인 (1.0 고정) |

D 유형 본인의 가치는 이미 Base = 100에 반영되어 이중 계산 방지.

## 4. Staged Scoring (3단계 산정)

| Stage | 상태 | 트리거 | 산식 적용 | 원장 기록 | KPI 5-B 분포 목표 |
|---|---|---|---|---|---|
| **1. Pending** | 임시 산정 | 기여 발생 즉시 | `Base × Difficulty` (Q/R/EP = 1.0) | `credit_issued{stage: pending}` | 60~75% |
| **2. Confirmed** | 백테스트 보정 | 해당 케이스 합의 확정 시 | Quality 갱신, Rule이면 Reusability 누적 시작 | `credit_adjusted{stage: confirmed, diff: ±X}` | 15~30% |
| **3. Audited** | 최종 확정 | 100건 종료 / 이의제기 / 30일 경과 | 관리자 검토 + outlier 조정 + ErrorPrevention 누적 마감 | `credit_adjusted{stage: audited, diff: ±X}` | 5~15% |

**무결성 규칙**:
- 모든 Stage 전환은 원장에 **신규 entry로 append**
- 기존 entry는 절대 수정·삭제 X
- 차액(`amount_diff`)만 새 entry로 기록 → 해시 체인 무결성 + KPI 5-A 추적률 100%

## 5. KPI 5-D 분포 시뮬레이션 (10명 × 100건 가정)

| 유형 | 추정 이벤트 수 | 평균 Credit/이벤트 | 합계 | 비중 |
|---|---|---|---|---|
| Draft | 1,000 (10명 × 100건) | 60 × 1.0 × 1.2 ≈ 72 | 72,000 | **47.7%** |
| Review | 900 (100건 × 9회) | 40 × 1.0 × 1.2 ≈ 48 | 43,200 | **28.6%** |
| Rule | 25 (추정) | 200 × 1.2 × 2.5 ≈ 600 | 15,000 | **9.9%** |
| ErrorPrevention | 200 (추정) | 100 × 1.0 × 1.0 = 100 | 20,000 | **13.2%** |
| **합계** | | | **150,200** | **100%** |

**판정**:
- 모든 유형 < 60% → KPI 5-D 통과 ✓
- 모든 유형 ≥ 9% → 4종 분류의 정량적 정당성 확보 ✓
- Rule·ErrorPrevention 합 23% → 차별점인 "지식 자산화"가 분포에 가시화됨

**보정 정책**: 본선 전 파일럿 결과에서 어느 유형이 60% 초과 시 Base 값을 ±20% 범위에서 보정. 보정 이력은 [KPI_정의.md](KPI_정의.md)에 timestamp 명시 추가.

## 6. 이의제기 (Dispute) 절차

```
[전문가] dispute_filed 이벤트 발행 (이유 명시 필수)
    ↓ 원장에 신규 entry로 append
[관리자] 검토 (SLA 7일)
    ↓
판정: accept / reject
    ↓
accept 시:
  - dispute_resolved{adjustment_amount, reason} entry 추가
  - credit_adjusted{stage: audited, diff: ±X} entry 추가
  - 원 credit_issued entry는 그대로 유지 (해시 체인 무결성)

reject 시:
  - dispute_resolved{accepted: false, reason} entry 추가
  - 크레딧 변경 없음
```

**KPI 5-C 산식 재정의**:
- `dispute_rate` = `dispute_filed / total_credit_issued` (목표 ≤ 10%)
- `dispute_acceptance_rate` = `dispute_resolved(절차 정상 처리) / dispute_filed` (목표 ≥ 95%)
- 여기서 "정상 처리"는 accept/reject 결정 + 사유 기록 + SLA 7일 내 처리. accept 비율 자체가 목표가 아님.

## 8. 본선 전 보정 미결 사항

1. **Base 값 검증**: 1·2개월차 파일럿 데이터로 KPI 5-D 분포 확인 후 ±20% 범위 조정
2. **Difficulty 가중치 (0.3/0.2/0.3/0.2)**: 회귀 분석으로 합의 시간·전문가 동의도와의 상관 검증
3. **Quality 트리거**: "큰 수정" / "관리자 우수 표시"의 기계적 정의
4. **Reusability 측정**: 규칙 "참조" 정의 — 직접 인용 / 의미 참조 / 후처리 검증 중 어느 단계까지 카운트 ([KPI_정의.md](KPI_정의.md) 미결 #4와 연동)
5. **ErrorPrevention 검출**: 자동판정이 과거 기여를 "참조"했음을 어떻게 로깅할지 — Consensus Engine의 RAG 호출 시 참조 source ID를 명시 기록
